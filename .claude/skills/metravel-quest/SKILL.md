# metravel-quest

Движок и регламент для публикации городских квестов на metravel.by.
Квест — это пеший маршрут по точкам города: на каждой точке игрок читает живую
историю/легенду и выполняет задание (что-то посчитать, найти, прочитать).

**Модель:** квесты пишутся ТОЛЬКО на opus (правило владельца) — sonnet
галлюцинирует «наблюдаемые» детали, которые механический чек не ловит.

## Редакционная модель: город = слои квестов

Каталог города наращивается слоями. Один город может (и должен) иметь
несколько квестов с РАЗНЫМИ сюжетами и минимальным пересечением точек:

1. **Слой «классика»** — первый квест города: главные must-see по рейтингам
   (Google Maps top sights, TripAdvisor), обёрнутые в главную легенду/историю
   города (Вавельский дракон, варшавская Сирена).
2. **Слой «неочевидный город»** — второй и дальше: то, что не видит турист с
   путеводителем. Проверенные жанры:
   - **муралы и стрит-арт** (эталон: minsk-dvoriki, vitebsk-teens-street-art-map);
   - **дворики, районы, слобóды** (krakow-kazimierz, gomel-spasova, minsk-traktorny);
   - **кино-локации** — «здесь снимали <фильм/сериал>»: точки = кадры, задания =
     сверь кадр с натурой (эталон темы: zheludok-palace/«Масакра»). Фильм должен
     быть реально снят на этих местах (пруфы: IMDb locations, интервью,
     краеведческие источники) — не «похожие места»;
   - **тематические**: персона (Шагал, Пикассо, Коперник), эпоха (авангард,
     соцгород), ремесло/еда (порто-вино, янтарь), мистика/легенды.
3. **Слой «детские»** — по возрастам 5-7 и 8-10 (см. раздел ниже).
4. **Слой «подростковые»** — 11-14 (см. раздел ниже).

Перед новым квестом города проверь существующие
(`GET /api/quests/` — пагинировано, обходи по `next`!) и выбери слой/сюжет,
которого ещё нет. Точки нового слоя не должны повторять точки старых квестов
города (допустимо 1-2 пересечения, если сюжет смотрит на них иначе).

## Принципы (ОБЯЗАТЕЛЬНО)

1. **Не дублировать.** Перед созданием проверь существующие квесты и не делай
   повтор по `quest_id` и по сюжету города:
   `GET https://metravel.by/api/quests/` (список, ПАГИНИРОВАН — обходи все
   страницы по `next`) и `GET /api/quests/by-quest-id/<quest_id>/` (деталь).
   Если квест с этим сюжетом уже есть — обновляй его (тот же `quest_id`,
   миграция идемпотентна), а не плоди дубль. Для второго квеста существующего
   города НЕ создавай новый quest-city: `POST /api/quest-cities/` не дедупит —
   переиспользуй существующий `city id` (universal-миграция дедупит по имени).
2. **8–12 точек в СТРОГО последовательном пешем маршруте — БЕЗ возвратов**
   (у детских меньше, см. возрастную сетку). Упорядочь точки как реальный пеший
   путь: выбери старт (вход в город/район, крайняя точка — напр. самый север),
   дальше каждый следующий шаг — ближайшая ещё не посещённая точка по ходу
   движения (greedy nearest-neighbor), маршрут монотонный по оси города
   (север→юг или последовательно по районам). Поле **`order` = фактический
   порядок прохождения**; фронт рисует шаги строго по `order`.
   **АНТИПАТТЕРН (так НЕ делать, это баг):** 1→2→3 идут на юг, потом шаг 4
   возвращает игрока на север к зоне точки 1, а шаг 5 снова гонит на юг — человек
   мечется по карте. Каждый переход между соседними по `order` точками не должен
   разворачивать игрока назад к уже пройденной зоне.
2a. **Маршрут = последовательный ИЛИ кольцевой (load-bearing).** Допустимы ровно
   две геометрии: (а) **последовательная** — точки идут цепочкой, игрок нигде не
   возвращается; (б) **кольцевая** — последняя точка приходит обратно к первой
   (удобно, если человек оставил машину/начал у вокзала; для детских в парке —
   предпочтительная форма). Любая другая форма — баг. Проверяй геометрию по
   координатам, а не на глаз.
2b. **Опциональные точки («по желанию»: кафе, корчма, музей) — ТОЛЬКО по пути и
   ТОЛЬКО в середине маршрута.** Привал нужен человеку в процессе прогулки, а не
   после финала, поэтому: (1) не ставь их в конец («сходи поешь, квест окончен» —
   бессмысленно) и не в начало; (2) ставь в ту позицию между обязательными
   точками, где крюк минимален — формально: `d(prev,opt) + d(opt,next) −
   d(prev,next)` минимально, и это значение **должно быть < 250 м**. (3) Если у
   выбранного кафе крюк ≥250 м при ЛЮБОЙ позиции — кафе просто не у маршрута:
   **бери другое, ближе к линии маршрута** (`node scripts/quest-poi-suggest.js
   --quest-id=<id> --kinds=cafe`), а не «пристраивай» его в хвост.
   Диагностика по живому квесту — сканер крюков (см. скилл `metravel-quest-geocheck`).
   В маршруте на 2+ часа обязательна хотя бы одна точка-привал (кафе/парк) в
   середине — человек должен иметь возможность «отойти покушать» не выпадая из
   сюжета.
3. **Квест — это СВЯЗНАЯ ИСТОРИЯ, а не список точек. Погружение.** У квеста
   сквозная нить: `intro` задаёт завязку и героя/тему, каждая точка двигает
   рассказ дальше, финал закрывает арку. Человек должен чувствовать себя ВНУТРИ
   истории, идти по сюжету, а не отмечаться на достопримечательностях. Между
   точками — мостики («отсюда переулок выведет тебя к…»), чтобы маршрут читался
   как одно повествование. Тема/мотив (легенда, личность, эпоха, ремесло,
   событие, фильм) проходит через весь квест.
3a. **На каждой точке — живой слой деталей.** Не сухая справка, а то, что
   делает место интересным: ЛЕГЕНДА (подавай как легенду), реальная ИСТОРИЯ и
   событие, ЛИЧНОСТИ (кто здесь жил/творил/бывал, человеческая деталь),
   АРХИТЕКТУРА (что видно глазами и почему так — стиль, материал, деталь),
   КУЛЬТУРНЫЕ ОСОБЕННОСТИ и быт, происхождение названия (топонимика). Всё
   ДОСТОВЕРНОЕ, сверенное веб-поиском (≥2 независимых источника на нетривиальный
   факт): факт не подтверждён — не пиши его, новую «наблюдаемую» деталь не
   выдумывай. Факты — как факты, легенды — как легенды.
4. **Игровое задание, выполнимое на месте.** На каждой точке — задание, которое
   реально сделать стоя перед объектом: найти деталь, прочитать табличку/дату,
   назвать предмет в руках статуи, узнать блюдо/материал/символ. Не абстрактные
   вопросы из интернета, а наблюдение здесь и сейчас. У каждого задания — `hint`
   и корректный `answer_pattern`. **Счёт объектов — ТОЛЬКО при 100% уверенности
   в числе** (устойчивый, не зависит от ракурса/сезона/реставрации). Число
   плавающее/спорное → не делай счёт «с запасным range», а замени на более
   ИНТЕРЕСНЫЙ вопрос с бесспорным однозначным ответом (эталон: Plac Nowy —
   вместо счёта окошек «что за блюдо жарят?» = запеканка). Где счёт оставлен —
   давай **мягкий `range` ±1** вокруг верного числа (n≥3, где реально сбиться:
   8 лягушек → 7–9, 12 кресел → 11–13); `exact` — только для бесспорных 1–2
   (2 башни, 1 проезд), где ±1 пропустил бы явно неверное.
4b. **Планка вопроса (4 критерия, ВСЕ обязательны):** (1) ответ 100% однозначен
   и проверяем (источник/видно на месте); (2) виден стоя на точке — наблюдение
   здесь-и-сейчас, не эрудиция; (3) НЕ выдаётся в `story`/`hint`; (4) объект
   стабилен (не снимут, не зависит от сезона/реставрации/ракурса).
4c. **Одобренные ТИПЫ вопросов** (опирайся на них): (A) **узнай культовую вещь
   места** — фирменный символ/блюдо/явление (запеканка на Plac Nowy);
   (B) **прочитай конкретную надпись/дату/имя** на табличке/памятнике/надгробии
   (сверь по фото/Mapillary, что читаемо); (C) **найди спрятанную стабильную
   деталь** и назови (бронзовый дракончик у моста); (D) **что символизирует /
   кто-что изображён** — смысл памятника (пустые стулья → депортированные) или
   узнаваемая фигура/зверь/предмет на рельефе/гербе/статуе. ИЗБЕГАЙ: счёт
   нестабильного (окна/гномы/утки/деревья/ярусы); архитектурные термины-загадки
   (аркада, аттик, пинакли); субъективное («сколько ярусов», «какая форма»);
   невидимое-эрудиция. Где НИ ОДИН тип не даёт 100% — делай свободный
   «впечатленческий» `any_text` (живо, погружает, всегда засчитывается), а НЕ
   шаткий вопрос.
4a. **Подсказка НЕ содержит ответа.** `hint` — это наводка «куда/как смотреть»
   («подойди ближе к стене», «подними взгляд на вершину колонны», «пересчитай
   колонны на фасаде»), а НЕ сам ответ и не его половина. Запрещено: называть
   ответ или его синоним («…— от СССР», «это гусеница танкетки Голиаф»), давать
   точное число/диапазон ответа («их от 3 до 5»), писать «Крест и …». Подсказка
   открывается только после 2 неверных попыток — она должна помочь додумать, а
   не выдать. Проверка перед заливкой: для каждого шага убедись, что ответ
   (и ключевые слова `answer_pattern.value`) не встречается в тексте `hint`.
5. **Вступление и финал.** `intro` (`is_intro`, `answer_pattern.type='any'`) —
   завязка, тон, правила прохождения, направление к первой точке. Финал
   (`quest-finales`) — тёплый человеческий вывод о городе и маршруте. Это
   ТЕКСТ финала; финальное ВИДЕО+постер — отдельный пост-шаг, см. скилл
   `metravel-quest-finale` (не забудь его для нового квеста).
6. **Тон живой, человеческий.** Без AI-обвязки, без канцелярита, без эмодзи и
   шаблонных заголовков. Пиши как увлечённый гид, а не как генератор.
7. **Интересные опциональные места С ИНФОРМАЦИЕЙ ДЛЯ ПОСЕТИТЕЛЯ.** По ходу
   маршрута добавляй ОПЦИОНАЛЬНЫЕ точки (`answer_pattern.type='any'`, пометка
   «по желанию») — то, что обогащает прогулку: музей (напр. Czartoryski с «Дамой
   с горностаем»), тематический/кошерный ресторан, знаковая кофейня, смотровая.
   Для каждого такого места заполняй **структурное поле `poi_info`** (см. раздел
   ниже) — часы работы, цена билета, сайт; фронт показывает это карточкой на
   шаге. Часы/цену из OSM/поиска СВЕРЯЙ с официальным сайтом (устаревают).
   Источник кандидатов — `node scripts/quest-poi-suggest.js --quest-id=<id>`
   (`--kinds=museum,gallery,kosher,cafe,attraction`). Опциональная точка не
   блокирует прохождение.

## poi_info — информация для посетителей (бэкенд-контракт)

У шага квеста есть JSON-поле `poi_info` (nullable). Заполняй его на:
- каждой точке-МУЗЕЕ или платном объекте (замок, башня, усадьба) — даже
  обязательной по сюжету;
- каждой ОПЦИОНАЛЬНОЙ точке (кафе, музей, смотровая) с расписанием/ценой.

Схема СТРОГАЯ — бэкенд-валидатор отклоняет всё лишнее:

```js
poi_info: {
    is_museum: true,                        // ОБЯЗАТЕЛЬНОЕ boolean-поле
    opening_hours: 'вт–вс 10:00–18:00',     // опц., непустая строка ≤500
    ticket_price: '15 BYN, детям до 7 бесплатно', // опц., непустая строка ≤500
    website: 'https://example.by/',         // опц., валидный http(s) URL ≤500
}
```

- Разрешены ТОЛЬКО эти 4 ключа (`unknown_fields` → 400). `is_museum` обязателен
  внутри объекта; для кафе/смотровой ставь `is_museum: false`.
- Обычный шаг без расписания — `poi_info` не передавай вовсе (null).
- Часы пиши человекочитаемо и по-русски («вт–вс 10:00–18:00, пн выходной»),
  цену — с валютой и детским тарифом, если есть.
- Данные УСТАРЕВАЮТ: часы/цену бери с официального сайта на момент написания;
  если объект сезонный/нестабильный — пиши осторожно («летом обычно…») или
  оставляй только website.
- Запись: universal-миграция `migrate-quest-from-file.js` и синк
  `sync-quest-to-prod.js` передают `poi_info` из data-файла как есть
  (POST/PATCH `/api/quest-steps/`); чтение — `poi_info` в bundle, фронт
  (`questWizardStepCard`) рисует блок «Часы работы / Билеты / Сайт».
- При обновлении СТАРЫХ квестов (до июля 2026): текстовые упоминания часов в
  `story` оставь (это часть рассказа), но продублируй структурно в `poi_info`.

## Детские (5-7, 8-10) и подростковые (11-14) квесты

Отдельный формат со своей механикой. НЕ «взрослый квест попроще», а другой
жанр: у ребёнка должна быть РОЛЬ в истории (помощник героя, детектив,
хранитель), у подростка — миссия без «детскости».

### Возрастная сетка (load-bearing)

| Тир | Теги (обязательные) | Точек | Длина | Время | Где |
|---|---|---|---|---|---|
| 5-7 | `kids`, `age-5-7`, `family`, `adult-required` | 6–8 | ≤1.5 км | 45–60 мин | ОДИН парк / пешеходная улица, без переходов проезжих частей |
| 8-10 | `kids`, `age-8-10`, `family`, `adult-required` или `adult-recommended` | 7–9 | ≤2.5 км | 60–90 мин | пешеходные зоны, минимум регулируемых переходов |
| 11-14 | `teens`, `age-11-14`, `adult-recommended` | 8–10 | ≤4 км | 90–120 мин | обычный городской маршрут |

Плюс тег безопасности маршрута: `public-park-walk` (весь квест в парке),
`public-walk` (пешеходные улицы), `public-art-walk` (муралы/скульптуры).
Тег `kids` включает бейдж «Детский» и фильтр детских на фронте
(`isKidsQuest` в `QuestsScreen.helpers.ts`) — без него квест не попадёт в
детскую подборку.

### Сюжет по возрасту

- **5-7 — сказка.** Логика мультика/сказки: волшебный герой-спутник (потерянная
  песенка, львёнок, убежавший рисунок), которому надо помочь. История слушается
  с голоса — тексты КОРОТКИЕ (story ≤ 600 знаков), читает взрослый вслух. Одна
  простая арка: потерял → ищем по знакам → нашли, праздник. Никаких дат, войн,
  смертей, страшного.
- **8-10 — приключение/детектив.** Миссия со сбором знаков: паспорт хранителя,
  семь голосов, бюро добрых дел. Ребёнок читает сам простые надписи, считает,
  сравнивает. Можно мягкую историю города (кто построил, зачем башня), нельзя —
  перегруз датами. Юмор работает.
- **11-14 — город как загадка, без сюсюканья.** Жанры: детектив по исчезнувшим
  зданиям, городской код/символы, стрит-арт как галерея, слои времени,
  урбанистика. Подача на равных, темы «как читать город». Интересный факт >
  назидание. Обращение на «ты», никакого «ребята» и «дорогой юный друг».
- **Привязка темы к городу обязательна.** Сказка/герой должны иметь причину
  быть ИМЕННО здесь: Хоттабыч в Витебске — потому что Лагин родился в Витебске;
  трубач Могислав — с ратуши Могилёва. Случайный мультик-франшиза без связи с
  городом — нельзя (плюс чужие бренды/персонажи под копирайтом не используем:
  свои герои по мотивам города, а не Эльза/Щенячий патруль).
- **Тематический парк/аттракционы** — допустимая площадка для 5-7/8-10
  (кольцевой маршрут по парку со сказочной легендой), но задания всё равно
  наблюдательные, не «купи билет на карусель».

### Задания по возрасту

- **5-7:** найди и покажи; посчитай до 5 (лапы, фонари, окошки — только
  стабильное); какого цвета; какой звук/жест повтори. Ответ вводит взрослый.
  Больше `any`/`any_text` (впечатление, «расскажи, что увидел»), проверяемых —
  2-3 на квест, числа малые.
- **8-10:** прочитай короткую надпись/имя; посчитай до 10-12 (мягкий range);
  найди деталь по описанию; сравни («что общего у этих двух фигур»). Собирай
  «знаки» по сюжету — каждый ответ что-то даёт истории.
- **11-14:** те же 4 одобренных типа вопросов, что у взрослых (см. 4c), плюс
  задания на интерпретацию: «что зашифровал художник», «найди, чем этот дом
  отличается от соседей». Можно двухходовые (найди → сопоставь).
- Планка достоверности ТА ЖЕ, что у взрослых: наблюдаемая деталь подтверждена
  источником/фото, ответ однозначен, hint без ответа. Детский квест — не
  повод выдумывать скульптуры (см. грабли: фейковый «кот» в Гродно).
- Роль взрослого пропиши в intro: «этот квест ребёнок проходит вместе со
  взрослым; читайте истории вслух» (5-7) / «взрослый рядом, но решает ребёнок»
  (8-10).

### Безопасность маршрута (детские)

- 5-7: маршрут ЦЕЛИКОМ внутри одного парка или пешеходной зоны. Ноль переходов
  проезжей части. Кольцо предпочтительно (закончить у входа, где начали).
- 8-10: переходы только по светофору/зебре, и о каждом предупреждай в story
  («за руку со взрослым переходите…»).
- Точки — в людных, просматриваемых местах; никаких заброшек, глухих дворов,
  парковок, кромок воды без ограждения.
- Опциональная точка-привал для детских — не кафе с алкоголем, а мороженое,
  детская площадка, туалет (для родителей это логистика №1) — с `poi_info`.

## Технический раздел

### Формат файла данных

По образцу `scripts/warsaw-quest-data.js` — `module.exports = [ { quest } ]`.
Структура одного квеста: `quest_id` (строковый, kebab-case), `title`,
`city: { name, lat, lng, country }`, `meta`, `storage_key`, `intro`, `steps[]`,
`finale`. У каждого шага — `step_id`, `title`, `location`, `story`, `task`,
`hint`, `answer_pattern: { type, value }`, `lat`, `lng`, `mapsUrl`,
опционально `poi_info` (см. контракт выше).

- **Координаты** — decimal degrees (`lat`, `lng`), напр. `52.25123, 21.00862`.
- **mapsUrl** — ссылка на Google Maps: `https://maps.google.com/?q=<место>`.
- **meta**: `duration_min` (минуты на прохождение), `difficulty`
  (`easy`/`medium`/`hard`), `tags` (см. таксономию), опц. `pet_friendly`.
- **country** — id страны квеста (НЕ travel-страна!): Польша=160, Беларусь=3,
  Армения=6, Литва=—, полный список `GET /api/countries/`.
- **quest_id детских:** `<city>-kids-<theme>` (5-7 и 8-10),
  `<city>-teens-<theme>` (11-14).

### Типы answer_pattern и когда какой

- `any` — `{ type: 'any', value: '' }` — любой ответ принимается. Для `intro`
  и шагов «просто дойди / нажми далее».
- `exact` — `{ type: 'exact', value: '<строка>' }` — точное совпадение после
  нормализации. Для одного известного числа/слова без вариантов.
- `exact_any` — `{ type: 'exact_any', value: JSON.stringify([...]) }` — массив
  допустимых вариантов (lowercase, без пунктуации). Перечисли падежи и синонимы:
  `['меч и щит','щит и меч','меч, щит',...]`. Для слова/названия/предмета.
- `range` — `{ type: 'range', value: JSON.stringify({ min, max }) }` — число в
  диапазоне. Для заданий «посчитай» (счёт может слегка разойтись).
- `any_text` — `{ type: 'any_text', value: JSON.stringify({ min_length }) }` —
  любой осмысленный текст не короче N символов. Для «опиши в нескольких словах».
- `any_number` — любое число. Когда важен сам факт числа, не его значение.
- `approx` — `{ type: 'approx', value: JSON.stringify({ target, tolerance }) }` —
  число около цели ± допуск.

Нормализация ввода (`buildAnswerChecker`): lowercase, схлоп пробелов, удаление
пунктуации, ё→е. Учитывай это, заполняя варианты `exact_any` в lowercase.

**Подсказки по выбору:** счёт объектов → `range`; известное точное число →
`exact`; слово/название → `exact_any` (перечисли падежи и синонимы); описание
своими словами → `any_text` с `min_length`; «дойди/нажми» → `any`.

NB: прод возвращает `answer_pattern` уже ОБЪЕКТОМ (не строкой) — конвертер
prod→data-file обязан это учитывать, иначе всё превратится в `type:'any'`.

## Процесс

1. **Выбери слой и сюжет** (см. редакционную модель): что у города уже есть,
   какого слоя не хватает, какая легенда/тема/фильм понесёт маршрут.
2. **Ресёрч** — собери точки слоя и их легенды/историю/топонимику через
   веб-поиск (≥2 источника на нетривиальный факт). Для музеев/платных — часы,
   цены, сайт (для `poi_info`). Упорядочь в пеший маршрут.
3. **Напиши данные** — `scripts/<city>-quest-data.js` (детские:
   `<city>-kids-quest-data.js`) по образцу `warsaw-quest-data.js`, с явным
   `answer_pattern` на каждом шаге и `poi_info` на музеях/опциональных.
3a. **Русское название страны (иначе в каталоге будет сырой код).** Каталог
   квестов группирует города по стране и берёт заголовок группы из
   `COUNTRY_NAMES` в `screens/tabs/QuestsScreen.helpers.ts`, а код страны —
   из `utils/geoCountry.ts` (`getCountryCodeByCoords` по координатам города).
   Для новой страны: (а) убедись, что её ISO alpha-2 код с bbox есть в
   `geoCountry.ts` — иначе добавь; (б) убедись, что этот код есть в
   `COUNTRY_NAMES` с русским названием — иначе добавь (напр. `GR: 'Греция'`).
   Регресс-страж: `__tests__/screens/QuestsScreen.helpers.test.ts`.
4. **Миграция** — универсальная: `node scripts/migrate-quest-from-file.js
   --source-file=scripts/<city>-quest-data.js --dry-run` (идемпотентна,
   city-reuse по имени, передаёт `poi_info`). Отдельный `migrate-<city>-quest.js`
   пиши только если нужна нестандартная логика. Квест создаётся со `status=1`.
5. **Dry-run** — прогони миграцию в режиме проверки: данные читаются, маппятся,
   нет ошибок.
5a. **Гео-сверка точек (до заливки).** Прогони
   `node scripts/quest-geocheck.js --source-file=scripts/<city>-quest-data.js --quest-id=<quest_id>`
   и разбери WARN/FAIL: каждая точка должна стоять на объекте из `story`/`task`,
   а не на парковке/остановке/дороге (см. скилл `metravel-quest-geocheck`).
   Исправь `lat/lng/mapsUrl` в data-файле и перепрогони до 0 реальных проблем.
   Nominatim: последовательно, `--delay-ms>=1500`, параллельные агенты ловят 429.
5b. **Проверка ПОРЯДКА маршрута (anti-backtracking, до заливки).** Выпиши
   координаты шагов в порядке `order` и убедись, что маршрут идёт связно, без
   разворотов (гео-чекер ловит и это: `--backtrack-m`, дефолт 150 м; у кольцевых
   и долинных маршрутов возможны ложные срабатывания 1D-оси — решает геометрия).
   Если нашёлся возврат — переставь `order` (greedy nearest-neighbor от старта).
   Тексты-переходы в `story`/`task` («иди дальше к…») сверь с НОВЫМ порядком.
5c. **Leak-чек.** Ответ (и ключевые слова `answer_pattern.value`) не встречается
   в `hint`, `story` этого шага и в story СОСЕДНИХ шагов (число из story шага 3
   не должно быть ответом шага 7).
5d. **Реордер существующего прод-квеста — БЕЗ потери прогресса.** Если правишь
   порядок у уже залитого квеста (люди могли его пройти), меняй **только поле
   `order`** у существующих шагов через `PATCH /api/quest-steps/<id>/` (двухфазно:
   сначала временные order 900+, потом финальные — во избежание коллизий unique).
   **НЕ меняй `step_id`, НЕ удаляй/не пересоздавай шаги, НЕ бампай `storage_key`**
   — прогресс прохождения ключуется по `step_id`. Точечные правки текстов/полей
   существующих шагов — `scripts/sync-quest-to-prod.js` (PATCH по step_id,
   поддерживает `poi_info`; `--reorder` только явно).
6. **Залей на прод:**
   `NODE_TLS_REJECT_UNAUTHORIZED=0 node scripts/migrate-quest-from-file.js --source-file=… --api-url=https://metravel.by`
   (токен из `~/.metravel_token` или `--token=`; если протух (HTTP 401) — свежий
   логином e2e: `METRAVEL_TOKEN=$(node scripts/get-quest-token.js) node scripts/…`).
7. **Проверь GET-ом** `GET /api/quests/by-quest-id/<quest_id>/`: на месте
   `intro`, все `steps`, `finale`, корректные типы `answer_pattern`, координаты,
   `mapsUrl`, `poi_info` на музеях/опциональных, теги (детские — возрастной тир).
8. **Обложку** — AI-арт: напиши `assets/quests/<camelCaseId>/PROMPT.md`,
   генерит владелец; заливка `upload-missing-quest-covers-prod.js` (НЕ
   require()-ить его для проверки — сразу исполняет main()).
9. **Финале-видео** — скилл `metravel-quest-finale` (finaleId == numeric quest id).
10. **Пройди квест залогиненным** на мобильном вьюпорте: понятны ли задания,
   выполнимы ли на месте, работает ли проверка ответов, рендерится ли блок
   посетительской информации из `poi_info`; проверь печатную версию
   (QuestPrintable). Счётчик прогресса на детали = только обязательные шаги
   (без `type=any`) — «0/5 при 8 шагах» не баг.
11. **Перелинковка со статьями.** Авто-блок «Квест по городу» на travel-детали
   срабатывает сам по координатам (`utils/questForLocation.ts`). Дополнительно
   поставь инлайн-ссылку из релевантной статьи-путеводителя Юли (author 1),
   если она есть: `scripts/seo-edit.js --append-file` (бэкапы в
   scripts/.seo-backups/); лид/SSG-мету статьи не трогать. Новые quest-URL уйдут
   в IndexNow кроном; в GSC «Запросить индексирование» — вручную владелец
   (квота ~11/день), ТОЛЬКО после пререндера статики на проде.

## Верификация

- Дев Expo Web проксирует `/api` на **препрод-бэкенд** — новый прод-квест там не
  виден. Для локального теста можно временно залить квест и на препрод
  (`--api-url=<preprod>`), затем удалить/перезалить на прод.
- **Prod Static** не имеет SPA-fallback для динамических роутов квестов
  (`/quests/{cityId}/{quest_id}`) — прямой заход по URL может не открыться,
  проверяй переходом из списка. URL детали бери из фактического `city_id`
  квеста (не угадывать — параллельные сессии сдвигают id).
- Прохождение квеста требует логина (AuthGate). Для QA логинься тестовым
  e2e-аккаунтом (см. CLAUDE.md), без ручного ввода пароля в поля.
- Транзиентный title «Квест не найден» до гидрации — норма.

## API-справка

- `GET /api/quests/` — список (ПАГИНИРОВАН: `results` + `next`).
- `GET /api/quests/by-quest-id/<quest_id>/` — деталь по строковому quest_id
  (intro + steps + finale + poi_info).
- Запись (Token, admin): `/api/quest-cities/` (НЕ дедупит по имени!),
  `/api/quests/`, `/api/quest-steps/` (принимает `poi_info` объектом,
  `geo_verify`), `/api/quest-finales/` (finale id == numeric quest id).
- Фронт-роут детали: `/quests/{cityId}/{quest_id}` — тянет по строковому
  `quest_id`. Проверка ответов — `utils/questAdapters.ts` (`buildAnswerChecker`);
  посетительская информация шага — `poiInfo` (`adaptPoiInfo`) в
  `components/quests/questWizardStepCard.tsx`.
