# metravel-design-audit

Сквозной аудит визуальной консистентности **всего сайта** — поиск расхождений дизайн-системы
между экранами и их фикс. Отличие от `/review-uiux` (точечно одна страница) — это полный
проход с акцентом на консистентность «страница ↔ страница». Не переделывать дизайн, а
выравнивать отклонения от существующей системы.

## Маршрут (порядок экранов)

1. Главная `/`
2. Список путешествий `/travels` (фильтры, поиск, карточки)
3. Детали путешествия `/travels/[slug]`
4. Карта `/map` + карточка точки
5. Квесты `/quests` + деталь квеста
6. Профиль `/profile`, настройки, публичный `user/[id]`
7. Статьи/article
8. Авторизация: `/login`, `/registration`
9. Статика: about, contact, 404

Каждый экран — в **390×844 (mobile)** и **1280×800 (desktop)**: screenshot + snapshot +
console_logs.

## Что проверять (оси консистентности)

- **Токены и тема:** одинаковые цвета/радиусы/тени/отступы из `DESIGN_TOKENS`/`METRICS`
  между страницами; нет хардкод-цветов; light+dark через `useThemedColors` корректны.
- **Типографика:** единые уровни `<Heading>`; нет разнобоя размеров/весов заголовков.
- **Карточки:** travel — только `UnifiedTravelCard`; фото-доминанта (~70% высоты, contain+blur,
  оверлеи в углах) одинаково на всех листингах; нет «голых» дублей карточек.
- **Шапки/тулбары:** на мобильном ≤20% вьюпорта, icon-only ряд (не вертикальный столбец);
  единый паттерн между экранами.
- **backdrop-blur:** на fixed/sticky барах мобильного — статичный frost `surfaceMuted`, не
  живой `backdrop-filter`; единообразно.
- **Touch-таргеты** ≥44×44 на mobile; **контраст** текста (серый-на-сером, текст на фото без
  подложки); нет горизонтального скролла на mobile.
- **Состояния:** loading (скелетоны, без прыжков/CLS), empty (понятный текст + действие),
  error (не белый экран, не сырой JSON) — единый визуальный язык на всех экранах.
- **Навигация:** активный таб/состояние, заголовок отражает контент, «назад» работает.
- **Язык:** UI-тексты на русском, без англицизмов.

## Шаги

1. Запусти preview, пройди маршрут в двух вьюпортах, собирая скрины/snapshot/console.
2. Сведи находки в **матрицу консистентности**: ось (токен/типографика/карточка/шапка/
   состояние) × экран → где отклонение от системы (со скриншотом-доказательством).
3. Severity: **P1** — ломает layout/блокирует сценарий; **P2** — заметное расхождение/трение;
   **P3** — полировка.
4. **Кодовый аудит точечно** по симптомам — делегируй `review-auditor` (hardcoded-цвета,
   фикс-ширины, a11y-лейблы) и `guard-enforcer` (нарушения гардов). Дрейф токенов —
   через `/metravel-design-system`.
5. **Чини** подтверждённое профильными агентами (travel/map/quest/profile-expert по зоне);
   каждую правку перепроверяй в браузере (скрин до/после mobile+desktop), прогоняй
   lint/typecheck/Jest scope.
6. **Отчёт:** матрица консистентности + список фиксов с доказательствами + что осталось (P3/
   бэклог — оформить через `/metravel-issue`).

## Правила

- Браузер-проверку делать самому, никогда не перекладывать на пользователя; не раскладывается
  превью — исчерпать альтернативу (Prod Static / Playwright) до блокера.
- Не репортить вкусовщину: каждая находка привязана к оси консистентности и видна на скрине.
- Дизайн не переизобретать — выравнивать к существующей системе.
- Mobile приоритетнее (основная аудитория).
- Один экран глубоко — `/review-uiux`; редизайн экрана — `/metravel-screen-redesign`.
- **Третья ось сравнения — устройство (эталон):** матрица консистентности включает не только
  mobile 390 × desktop 1280, но и native-приложение на устройстве (`adb exec-out screencap`).
  Mobile web обязан выглядеть как приложение: без web-only serif на мобильном, без hover-only
  элементов, без `Platform.OS`-ветвлений, скрывающих контент, темизация только через
  `useThemedColors()` (статичные `DESIGN_TOKENS.colors.*` на native не темнеют). Расхождение
  mobile web ↔ устройство — минимум P2, фикс в сторону устройства (см. правило в CLAUDE.md).

## Паритет mobile web ↔ устройство (обязательное правило)

«Мобильная версия» = mobile web (~390px, `isMobile`) + Android + iOS ОДНОВРЕМЕННО: пользователь на всех трёх должен видеть один и тот же дизайн. Когда в задаче сказано «мобильный/mobile» — это всегда все три платформы сразу, не только web.

- **Эталон — устройство.** Android/iOS-приложение оттестировано и принято как образец: при любом расхождении mobile web правится под устройство, НЕ наоборот.
- **Верификация UI-правок — на обеих платформах со скринами:** web-превью 390px (`preview_resize` + `preview_screenshot`) И устройство/эмулятор (`adb exec-out screencap -p`; dev-client сидит на том же Metro — HMR обновляет обе стороны).
- **Запрещены web-only визуальные ветвления в мобильном вьюпорте:** serif-шрифты и hover-only элементы — только desktop (`!isMobile`); контент-элементы (чипы, бейджи, кнопки) не скрывать через `Platform.OS === 'web'`, если на устройстве они видны.
- **Темизация:** для тематических поверхностей только `useThemedColors()` — `DESIGN_TOKENS.colors.*` на native это статичный светлый fallback, на web — живые CSS-переменные.
- **Попапы/карточки точек на картах** — один общий компонент на всех страницах и платформах (различия — только добавочный функционал), компактный, вся информация видна без обрезания по X и Y.
