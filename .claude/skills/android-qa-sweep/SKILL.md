# Android QA Sweep — пайплайн план → тест → борд → фикс → верификация (macOS)

## Предусловия
1. Физическое устройство по USB, отладка включена. adb автоопределяется (`which adb` или `~/Library/Android/sdk/platform-tools/adb`); проверка — `adb devices -l`. (`brew install android-platform-tools`, если нет.)
2. Dev-client `by.metravel.app` установлен, Metro запущен (`npm start`), порт проброшен: `adb reverse tcp:8081 tcp:8081`.
3. **Dev API-гейт (load-bearing!)**: `.env` обычно держит `EXPO_PUBLIC_API_URL=http://127.0.0.1:8085` (локальный бэк). Если бэк не поднят — приложение покажет **0 данных везде** (пустые списки/карта, LogBox «fetch failed»), и это легко принять за UI-баг. Для QA временно укажи прод-API и верни обратно:
   ```bash
   cp .env /tmp/env.backup
   sed -i '' 's|EXPO_PUBLIC_API_URL=http://127.0.0.1:8085|EXPO_PUBLIC_API_URL=https://metravel.by|' .env
   # ... QA ...
   cp /tmp/env.backup .env        # ОБЯЗАТЕЛЬНО восстановить
   ```
   Менять `.env` → нужен перезапуск Metro `--clear` + Reload бандла (EXPO_PUBLIC_* инлайнятся при сборке).
4. В dev-client отключена плавающая кнопка DevTools (dev-меню → toggle «Tools button») — иначе перекрывает гамбургер.
5. Пользователь залогинен (e2e-аккаунт из `.env.e2e` через программный логин; пароль руками не вводить). Незалогиненный — ок для публичных экранов (карта/статьи/квесты).

## Инструменты — `artifacts/android-qa/qa.sh`
Source-аем и вызываем функции (Capitalized). Из Bash-тула:
`cd <repo> && source artifacts/android-qa/qa.sh && Nav 'about' 6 && Shot '20-about'`

- `Shot "имя" [delay]` — screencap → даунскейл `sips -Z 1200` (>1200px = риск лимита Read-картинок). Карты-WebView грузятся 4–10с → delay 8+.
- `Crop "имя" [offY] [h]` — **полноразмерный** кроп (1080-wide, без даунскейла ширины) для ТОЧНОГО прицеливания по мелким маркерам Leaflet. Даунскейл-скрины врут по координатам.
- `Nav "route" [delay]` — deeplink `metravel://<route>`. Scheme из app.json (`metravel` с 2026-06-13; старее — `myapp`, `export QA_SCHEME=myapp`).
- `Tap x y`, `Swipe x1 y1 x2 y2 [ms]`, `Back`, `Menu` (RN dev-меню), `Pinch` (zoom-out двумя одновр. свайпами), `TypeText "латиница"` (кириллицу adb НЕ вводит).
- `Clickables [имя]` — uiautomator-дамп кликабельных, печатает label + bounds + **готовое `-> tap X Y`**. Координаты тапов брать ТОЛЬКО отсюда. Кириллица выводится корректно (UTF-8 из файла).
- `Logcat` — `ReactNativeJS:E` (краши/стектрейсы). `Focus` — текущий foreground-экран.
- Переопределение: `export QA_OUT=...` (по умолч. `/tmp/android-qa`), `QA_PKG`, `QA_SCHEME`.

## Ловушки (выучено на практике)
- **Лимит Read-картинок**: полноразмерные native-скрины (1080×2400) превышают лимит при многих картинках в истории. `Shot` даунскейлит до 1200; для прицеливания по маркерам — `Crop` (узкая полоса, остаётся <2000px). Серии до ~30 скринов или делегируй оценку субагенту (android-expert) со свежим контекстом.
- **Leaflet-WebView невидим для uiautomator**: маркеры/попапы внутри карты в дерево `Clickables` НЕ попадают — целиться только по `Crop`, проверять результат скрином, не дампом.
- **Тап по маркеру карты** (native) ОТКРЫВАЕТ нижнюю карточку места (`MapPlaceBottomCard`) — попадай по пину и ТЕСТИРУЙ карточку целиком (фото, действия Страница/Сохранить/♥/＋, навигация в Google/Organic/Waze, фуллскрин-фото, и обязательно — закрытие во всех состояниях, см. чеклист п.10-12). [ИСТОРИЧЕСКОЕ, перепроверять: старая заметка #223/F-46 утверждала «тап только рецентрит, карточку не открывает» — это УСТАРЕЛО, карточка открывается; именно эта стейл-заметка спрятала класс багов #497.]
- **АНТИ-СТЕЙЛ-ЗАМЕТКА (load-bearing):** заметка вида «известный баг: действие X ничего не делает / недоступно» — НЕ повод пропускать взаимодействие X. Заметки устаревают; «инертный» элемент мог стать активным и принести НОВЫЕ баги. Всегда выполняй взаимодействие сам и сверяй с реальностью, а не с заметкой. (Реально: вера в стейл-#223 «маркер карточку не открывает» → я не тапал маркеры → пропустил весь #497.)
- **Deeplink с query-параметрами** (`map?lat=&lng=&radius=`) применяется только на ПЕРВОМ монтировании экрана; на уже открытом табе игнорируется (mount-only capture). Чтобы применить — свежий старт приложения, не переход по табу.
- **force-stop роняет dev-client в лаунчер** (Metro-бандл не embedded). После `am force-stop` приложение НЕ откроется в прежнее место — нужно заново подключить бандл: запустить Metro → deeplink `metravel://expo-development-client/?url=http%3A%2F%2Flocalhost%3A8081` или тапнуть сервер в «Recently opened». Не force-stop'ай без нужды.
- **Reload бандла** после правки `.env`/кода: `Menu` (`adb shell input keyevent 82`) → тап «Reload» (или Nav на dev-client URL). Deeplink на тот же экран JS НЕ перезагружает.
- **СТЕЙЛ-БАНДЛ — главная причина «фикс не сработал» при device-проверке.** Fast Refresh часто НЕ подхватывает НОВЫЕ файлы/импорты (а иногда и правки) — на устройстве крутится старый код, и баг «не исправлен», хотя в репо всё верно. Перед проверкой ЛЮБОГО фикса делать явный Reload (keyevent 82 → «Reload», ждать ~12с пересборки) и УБЕДИТЬСЯ скрином, что новый UI реально появился (изменённый текст/компонент), а не старый — только потом тестировать поведение. Не помечать native-фикс «verify pending», когда устройство подключено (см. правило 7 в `docs/NATIVE_COMPAT_RULES.md`).
- **Чипы-фильтры переупорядочиваются** при выборе (выбранные уходят в начало ряда) — позиции прыгают, перечитывай `Clickables` после каждого тапа.
- **Pinch** = два `adb input swipe ... &` + `wait` (одновременно). Иногда срабатывает как пан — повтори; для пана карты свайпь по ЧИСТОЙ области (не по строке поиска y≈700).
- **LogBox-тост** лежит поверх таб-бара и перехватывает тапы по нижним табам — закрывать перед тапами по табам. Деталь ошибки = тап по тосту → стектрейс с причиной (часто прямо называет проблему, напр. API URL).
- Радиокнопки/тогглы иногда срабатывают со 2-го тапа. `uiautomator dump` может вернуть stale-XML — перепроверять скрином.

## Надёжные хелперы qa.sh (P0) — предпочитать координатам
- `TapLabel "текст"` — тап по элементу по тексту/accessibilityLabel (центр bounds), БЕЗ ручных координат и путаницы tap/Tap. Для табов/кнопок/меню — всегда это.
- `DumpTexts` — весь видимый текст экрана (дешёвая текстовая альтернатива чтению картинки; для большинства проверок скрин не нужен).
- `WaitText "текст" [timeout]` / `AssertText "текст"` — дождаться/проверить текст (убивает тайминги с асинхронными данными — не снимай скрин до резолва).
- `Reload [sec]` — dev-меню → тап «Reload» по лейблу. `ColdStart [sec]` — force-stop → relaunch → выбрать dev-server (сбрасывает Zustand-сторы; нужен когда правка в сторах/контексте/persist не подхватывается Reload).

## Runtime-состояние без правок кода (P1) — `[QA-STATE]` / `__QA__()`
Dev-хук `utils/qaDebug.ts` (встроен в `app/_layout`) логирует в `ReactNativeJS` тег `[QA-STATE]` на каждой смене роута: `{route, isAuthenticated, userId, favorites, history, apiUrl}`. Читать: `adb logcat -d ReactNativeJS:V '*:S' | grep QA-STATE`. Это снимает необходимость вставлять временный `console.log` для диагностики (как было в саге с полками). Ручной дамп — глобал `__QA__()`. Если нужно новое поле — добавить в `dumpQaState`.

## ПРАВИЛО №0 (load-bearing): СКРИН + UI/UX-оценка на КАЖДОМ экране и после КАЖДОГО действия
- На каждом экране и после каждого значимого действия — `Shot` (карты/тяжёлое — delay 8+), затем **критически оценить UI/UX по скрину**, а не только «открылось/нажалось».

- **Каждую находку — сразу в действие:** мелкое/однозначное чинить тут же и ре-верифицировать скрином; крупное/спорное — тикет на борд (area front/android, спринт «Android Release»). НЕ оставлять находку «на потом» без тикета.
- Серию скринов для UI/UX-оценки можно делегировать свежему субагенту (передать пути PNG) — он вернёт находки, ты фиксишь/тикетишь.

## Обязательный чеклист интеракций (LOAD-BEARING — эти классы РЕАЛЬНО пропускались, проверять ВСЕГДА)
Эти проверки обязательны на КАЖДОМ релевантном экране — статичного скрина и «кнопка нажимается» НЕДОСТАТОЧНО:
1. **Toggle-действия в ОБЕ стороны.** Любой тоггл/статус тестировать И включением, И выключением — это РАЗНЫЕ кодовые пути, выключение падает независимо. Обязательно: избранное (добавить **и убрать**), подписка (follow **и unfollow**), план/статус поездки (поставить **и убрать**), лайк, mark/unmark. (Реальный баг: «не убирается из избранного, пишет ошибку» — add работал, remove падал.)
2. **Скролл КАЖДОГО списка до САМОГО конца + последний элемент.** Доскроллить до низа и убедиться, что ПОСЛЕДНЯЯ карточка/строка полностью видна НАД таб-баром (BottomDock ~56px+safe-area), а не спрятана за футером. Любой full-screen список на табе должен иметь `paddingBottom` под док. (Реальный баг: рулетка — последняя карточка за футером.)
3. **Высота шапок ≤ ~20% viewport + фильтры всегда доступны.** Фиксированная шапка (заголовок+фильтры+легенда) не должна занимать >20% высоты экрана; фильтры/контролы должны быть ДОСТУПНЫ всегда (sticky), а не уезжать со скроллом. Замерять по скрину (низ фикс-шапки vs высота экрана). (Реальные баги: календарь — шапка ~38%; Места — фильтры уезжают.)
4. **Фидбэк действия = смотреть на ошибку.** После КАЖДОГО действия читать `Logcat` (`ReactNativeJS:E`) и экран на error-тост/красный LogBox. Действие может «визуально пройти», но залогировать ошибку (успешный remove → тост «Ошибка» из-за упавшего refresh; автосейв формы → `ReferenceError: Property 'Event' doesn't exist`).
5. **Формы: реально ВВОДИТЬ и СОХРАНЯТЬ** (тест-аккаунт, латиница; пробелы в `adb input text` через `%s`). Краши web-глобалов (`Event`/`window`/`document` без `typeof`-гварда) и автосейв-падения вылезают ТОЛЬКО при реальном вводе/сохранении, не при просмотре. Подозреваемые места ищет скилл `android-native-audit`.
6. **Клавиатура.** Открыть клавиатуру на каждом тексто-вводе (особенно чат/комментарии/визард) и убедиться, что поле ввода видно НАД клавиатурой и не за таб-баром (adjustResize + абсолютный док взаимодействуют — поле уезжает).
7. **Реальные действия РАЗРЕШЕНЫ и ОБЯЗАТЕЛЬНЫ с тест-аккаунтом.** sergey@lyte.com (`.env.e2e`) может создавать/удалять созданное/лайкать/слать сообщения (получатель — ignatieva_julia@tut.by) — НЕ пропускать эти сценарии из «осторожности»; после теста прибрать (удалить созданное, вернуть тогглы). Деструктив над ЧУЖИМИ/прод-данными вне тест-аккаунта — нельзя.
8. **Нажать КАЖДОЕ вторичное/icon-действие — топ-бары, overflow (⋯), плавающие контролы.** Самый пропускаемый класс: иконки в шапке деталки и под `⋯` — их легко не заметить, потому что они не главная кнопка. Обязательно нажимать: **скачать / экспорт / офлайн-точки / «открыть в приложении» / поделиться / печать / аудио-озвучка / сброс**. Native-only пути (файловая система, share-intent, Linking, expo-модули) часто падают там, где web-путь работает — поэтому «работает на web» НЕ закрывает сценарий, его надо нажать на устройстве и прочитать `Logcat`. (Реальный баг: квест → скачать/подготовить офлайн-точки → тосты «Не удалось подготовить/открыть точки» — legacy `expo-file-system` бросает в рантайме SDK 56; web-путь работал, поэтому отчёт «всё ок» был ложным.)
9. **Аудит целостности вёрстки на КАЖДОМ скрине — критически смотреть, не «нажимается ли», а «не развалилось ли».** Флагить как баг: текст усечён в мусор (локация → «.:..», «Ита.…», заголовок → двухстрочный перенос в узкой колонке); элемент обрезан краем экрана или перекрыт плавающим контролом (напр. аудио-плеер налезает на карточку); пустой переразмеренный блок (большая дыра без контента); карточка/контейнер шире вьюпорта или контент срезан фикс-высотой (`overflow:hidden`); FAB/док перекрывает метаданные нижней карточки; блок «разъезжается» при другой высоте/плотности. Оценивать по РЕАЛЬНОЙ ширине устройства; одного «кнопка есть» недостаточно. Эту визуальную оценку каждого скрина делать критическим взглядом (или делегировать свежему субагенту-ревьюеру по серии скринов) — именно она ловит то, что adb-дамп не видит.
10. **ЗАКРЫВАЕМОСТЬ каждого popup / sheet / modal / карточки / fullscreen-overlay — в КАЖДОМ его состоянии.** Любой не-инлайн оверлей ОБЯЗАН закрываться из UI. На каждом проверять ВСЕ способы дисмисса: кнопка ✕ И свайп-вниз И тап-вне (backdrop) И системный back. Системный back — это ФОЛБЭК, а НЕ «закрывается»: если убрать оверлей можно только аппаратным back — это баг (юзер не знает про back). И главное: сначала РАСКРЫТЬ все под-секции оверлея («Ещё»/expand/аккордеон), ПОТОМ пробовать закрыть в раскрытом виде — close-кнопка часто ломается/уезжает за пределы экрана, когда карточка растёт. (Реальный баг #497: ✕ карточки места — no-op на native, свайп отключён, фуллскрин-фото не закрывалось, и при «Ещё» ✕ уезжала за статус-бар → закрыть можно было только системным back.)
11. **ВСЕ СОСТОЯНИЯ компонента, не только дефолтное.** Каждый компонент гонять во всех состояниях: компакт/раскрыт, свёрнут/открыт, пусто/заполнено, обычный/fullscreen, мало контента/много. Фикс или фича, проверенные ТОЛЬКО в дефолтном состоянии, пропускают баги в остальных. Особенно после своего фикса — не «сработала ли моя кнопка», а «работает ли во всех состояниях, не сломалось ли соседнее».
12. **Overlay / z-order: при открытии оверлея проверить, что нижние плавающие контролы СКРЫТЫ и не просвечивают сквозь него.** Когда поверх карты/списка открывается карточка/sheet/modal — нижние тулбары/FAB/иконки контролов должны прятаться или быть перекрыты, а не «светить» поверх контента оверлея. Смотреть на скрин: нет ли иконок фона на карточке. (Реальный баг #497: верхний тулбар карты — фильтры/слои/список — рисовался поверх фото карточки места, потому что гейтился только на `sheetState!=='full'`, а не на `hasSelectedPlace`.)

Каждый фикс верифицировать повторным скрином на устройстве; тикет закрывать только после верификации.

Узкая пост-фикс проверка = повторный пропуск.
6. **Регресс**: `npm run test:run` + `npm run typecheck` + `npm run lint` до зелёного.

## Паритет mobile web ↔ устройство (обязательное правило)

«Мобильная версия» = mobile web (~390px, `isMobile`) + Android + iOS ОДНОВРЕМЕННО: пользователь на всех трёх должен видеть один и тот же дизайн. Когда в задаче сказано «мобильный/mobile» — это всегда все три платформы сразу, не только web.

- **Эталон — устройство.** Android/iOS-приложение оттестировано и принято как образец: при любом расхождении mobile web правится под устройство, НЕ наоборот.
- **Верификация UI-правок — на обеих платформах со скринами:** web-превью 390px (`preview_resize` + `preview_screenshot`) И устройство/эмулятор (`adb exec-out screencap -p`; dev-client сидит на том же Metro — HMR обновляет обе стороны).
- **Запрещены web-only визуальные ветвления в мобильном вьюпорте:** serif-шрифты и hover-only элементы — только desktop (`!isMobile`); контент-элементы (чипы, бейджи, кнопки) не скрывать через `Platform.OS === 'web'`, если на устройстве они видны.
- **Темизация:** для тематических поверхностей только `useThemedColors()` — `DESIGN_TOKENS.colors.*` на native это статичный светлый fallback, на web — живые CSS-переменные.
- **Попапы/карточки точек на картах** — один общий компонент на всех страницах и платформах (различия — только добавочный функционал), компактный, вся информация видна без обрезания по X и Y.
