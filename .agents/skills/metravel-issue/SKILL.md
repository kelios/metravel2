# metravel-issue

Регламент заведения **одного** бага или фичи в общепринятом формате issue-репорта
сообщества (GitHub `bug_report.md` / `feature_request.md`) с привязкой к процессу
MeTravel. Источник правды о задачах — общий MCP task board (Django `task_board` на
metravel.by); см. `docs/TASK_BOARD_MCP.md`. Операции с бордом — только через агента
**ticket-board**. Этот скилл НЕ пишет код фичи и НЕ чинит баг — он корректно его
**описывает и заводит**; реализация идёт отдельно (`/ticket-flow`, профильные агенты).

`$ARGUMENTS` — свободное описание проблемы/идеи от пользователя (может быть сырым).

## Что даёт шаблон

Шаблон сообщества отвечает на три вопроса, которые ты упоминаешь: **в чём проблема**,
**что должно происходить (что ждать)** и **как починить / куда копать**. Не выдумывай
свою структуру — используй один из двух файлов рядом:

- баг → `bug-report.md`
- фича → `feature-request.md`

Оба заканчиваются блоком `## Task Contract` — для `area=front`/`area=back` он обязателен
(без него тикет не готов к `todo` и не закрывается в `done`, см. `docs/TASK_BOARD_MCP.md`).

## Шаги

1. **Классифицируй** (`kind`) по рубрике из `docs/TASK_BOARD_MCP.md`:
   - `bug` — поломка существующего поведения (краш / 5xx / неправильно / визуальный глюк /
     усечение / регрессия / XSS). Перф-жалоба на конкретную поломку («эндпоинт медленный,
     502») = `bug`.
   - `feature` — новая возможность / экран / редизайн / новая интеграция / новый эндпоинт.
   - `task` — chore / рефактор / перф-ПРЕДЛОЖЕНИЕ / SEO / контент / тулинг / тесты. Это
     не issue-репорт — оформляй по `tasks/000-template.md`, не этим скиллом.

2. **Определи `area`** — `front` / `back` / `android` / `ios`. Баг бэка (Django/DRF/PostGIS,
   репо `../metravel-backend`) этот репо не чинит — только заводит `area=back` для владельца.

3. **Собери факты, не догадки.** Заполняй поля шаблона реальными данными:
   - для бага — точные шаги, реальный URL/экран, платформа, ветка/коммит, роль аккаунта,
     console/network/скрин как доказательство, статус-коды;
   - для фичи — конкретная боль пользователя, наблюдаемый результат, in/out of scope.
   Если критичных данных не хватает (нет шагов воспроизведения у бага) — задай ОДИН
   уточняющий вопрос пользователю, не выкладывай полупустой репорт.

4. **Воспроизведи баг сам, если это web и доступно** (preview-инструменты / Playwright) —
   подтверждённое воспроизведение и доказательство кладутся в секцию «Доказательства».
   Браузер-проверку не перекладывать на пользователя (правило проекта).

5. **Скопируй нужный шаблон**, заполни все секции, заполни `Task Contract`
   (Scope / User-visible result / Data-API contract / Dependencies / Fallback-mock policy /
   Validation / Done gate) проверяемыми фактами. Пустые плейсхолдеры = задача не готова.

6. **Заведи на борд через `ticket-board`** (`metravel_task_create`):
   - `title` с префиксом источника, который проставит борд (`[FE-…]`/`[BE-…]`/…);
   - `description` = заполненный шаблон целиком;
   - `kind` = bug/feature, `area`, `urgency`;
   - `sprint` обязателен — активный спринт нужного `area` (`metravel_sprints_list` →
     статус `active`); без спринта тикет невидим в планировании;
   - `reporter` — кто завёл.
   Если MCP-борд недоступен — допустим временный локальный draft в `tasks/` для
   последующего import/sync и удаления (см. `tasks/README.md`), но это исключение.

7. **Проверь человеческий шаг** (см. ниже) — если в работе есть ручное действие, вынеси
   его в отдельную задачу `needs_human=true` и свяжи с основной.

8. **Верни пользователю**: id тикета(-ов) на борде, kind/area/sprint и краткое резюме.

## Человеческий шаг (`needs_human`)

Тег `needs_human=true` — это маркер задачи, состоящей **исключительно из действий
человека**: того, что агент физически не может сделать (ручной шаг в Google Search
Console, апрув/решение владельца, заливка секрета, действие в стороннем кабинете,
проверка на физическом устройстве, публикация в сторе). Правила:

- **Не смешивать исполнителей.** Одна задача = один тип исполнителя. Агентская работа
  (код/фикс/тесты/деплой-команда) `needs_human` **никогда** не несёт.
- **Выноси, а не помечай.** Если внутри агентской задачи всплывает человеческий шаг — он
  становится **отдельной** задачей `needs_human=true`, а не флагом на смешанной задаче.
- **Пиши для человека.** Тело такой задачи (шаблон `human-task.md`) — простым языком:
  что сделать → зачем → шаги → где → готово когда. Без агент-механики и Task Contract:
  человек, открыв задачу, должен сразу понять, что от него требуется.
- **Связывай две задачи.** Через `ticket-board`:
  - агент не может продолжить без человека → агентская задача `blocked_by_id`/
    `depends_on_ids` указывает на человеческую (статус агентской — `blocked_by`);
  - человеческий шаг не блокирует, а лишь сопутствует → `related_to_ids` (мягкая связь).
- `area`/`sprint` человеческой задаче тоже обязательны; `kind` обычно `task`.

## Границы

- Один вызов = один баг/фича. Несколько проблем — несколько тикетов.
- Контент-задачи/рефактор/тулинг (`kind=task`) идут не сюда, а в `tasks/000-template.md`.
- Прогон по пайплайну (реализация→тест→ревью→релиз) — скилл `/ticket-flow`, не этот.
