# Tasks — формат и правила

Каталог `tasks/` хранит структурированные задачи (TASK-файлы) для бэкенда, SEO,
кросс-командной работы и всего, что нужно довести через цикл
discovery → plan → implement → validate → review → release.

Каждая задача — отдельный markdown-файл по шаблону `tasks/000-template.md`.
**Не выдумывай свою структуру** — копируй шаблон и заполняй поля. Создание/обновление
задач автоматизируют агент `task-author` и команда `/task-new`.

---

## Имя файла и ID

- **Имя файла:** `NNN-kebab-slug.md`, где `NNN` — глобальный сквозной счётчик с ведущими
  нулями (`001`, `002`, …, `010`). Бери `max(существующих NNN) + 1`.
- **Заголовок внутри файла:** `# TASK-YYYYMMDD-NNN: Short Title`, где
  - `YYYYMMDD` — дата создания,
  - `NNN` — тот же номер, что в имени файла,
  - `Short Title` — короткий человекочитаемый заголовок (англ.).
- `slug` в имени файла — короткая суть на kebab-case (`comment-delete-superuser-forbidden`).

Пример: файл `009-comment-delete-superuser-forbidden.md`, заголовок
`# TASK-20260608-009: Superuser cannot DELETE another user's comment (403)`.

## Шапка (метаданные)

```
Status: Backlog
Owner: Backend
Support: Frontend Developer, Tester, Reviewer, Releaser
Created: 2026-06-08
Updated: 2026-06-08
```

- **Status** — жизненный цикл: `Backlog → In Progress → In Review → Done` (и `Blocked`,
  `Cancelled` при необходимости). Когда задача закрыта, фиксируй верификацию:
  `Status: Done (verified YYYY-MM-DD)` и при желании цитату-проверку строкой `> ...`
  сразу под шапкой (см. `006-quests-sitemap.md`).
- **Owner** — главная роль-исполнитель: `Backend`, `Frontend`, `SEO`, `Manager`,
  `Content` и т.п. Конкретный исполнитель/агент — в секции `## Assignment`.
- **Support** — вспомогательные роли через запятую (`Developer, Tester, Reviewer, Releaser`).
- **Created** — дата создания, не меняется. **Updated** — дата последнего изменения файла,
  обновляй при каждой правке.
- Все даты — `YYYY-MM-DD`, абсолютные (не «вчера/на той неделе»). Текущую дату бери из
  контекста сессии.

## Секции (обязательны все, по порядку)

1. **## Goal** — 1–2 предложения о желаемом результате. Без «как», только «что и зачем».
2. **## Context** — фон, запрос пользователя, ссылки, файлы, ограничения, результаты
   probe/воспроизведения. Внутри — блок `Source task:` со ссылкой на источник:
   ```
   Source task:

   - Source id:        TASK-… / bug-id / тикет (если задача порождена другой)
   - Source path:      путь к исходному файлу/обсуждению
   ```
   Если источника нет — оставь поля пустыми, но **блок не удаляй**.
3. **## Acceptance Criteria** — чеклист `- [ ]` проверяемых критериев приёмки. Каждый
   пункт — наблюдаемый факт, а не задача.
4. **## Gherkin Tests** — один или несколько `Feature/Scenario` в блоке ```gherkin.
   `Given/When/Then` описывают наблюдаемое поведение, а не реализацию.
5. **## Assignment** — `Primary owner:` (кто/какой агент ведёт) и `Support agents:`
   (кто помогает, с указанием за что). Здесь маппинг ролей на реальных агентов проекта.
6. **## Likely Files Or Areas** — список файлов/модулей/эндпоинтов, которых задача
   касается. Для бэкенд-задач (отдельный репо) — логические области, не пути этого репо.
7. **## Plan** — нумерованные шаги реализации.
8. **## Validation** — конкретные команды или ручные проверки (`npm run …`, `curl …`,
   шаги в браузере). Не «проверить, что работает», а как именно.
9. **## Release Checklist** — чеклист готовности к релизу (changed files перечислены,
   новые файлы помечены, secret/cache/local исключены, staged по запросу, блокеры
   записаны). Шаблонные пункты не удаляй — отмечай по факту.
10. **## Progress Log** — хронология `- YYYY-MM-DD: что произошло.` Первая запись —
    `Created.`. Дописывай новые строки, старые не переписывай.
11. **## Results** — итог: `Changed files:`, `Validation evidence:`, `Reviewer findings:`,
    `Release notes:`, `Blockers:`. Заполняется по мере выполнения; до старта пусто.

## Роли → агенты проекта

Шаблон использует обобщённые SDLC-роли. Маппинг на реальных исполнителей:

| Роль в шаблоне | Кто исполняет |
|---|---|
| Manager / Primary owner | оркестратор (основной агент) |
| Developer (FE) | `travel-expert`, `map-expert`, `metravel-seo-expert`, `refactor-surgeon`, `dev-loop` |
| Developer (BE) | задача отдельного бэкенд-репо → выдаётся чеклистом, кодом тут не делается |
| Content / SEO | `travel-writer`, `metravel-seo-expert`, `index-doctor` |
| Tester | `test-author` |
| Reviewer | review-проход (`/code-review`, `$metravel-code-reviewer`) |
| Releaser | preflight + deploy по явному target env |

Бэкенд-задачи (отдельный сервис) не реализуются в этом репо — они оформляются TASK-файлом
и передаются владельцу/бэкендеру чеклистом из `## Results` / `docs/BACKEND_WORKBOARD.md`.

---

## Принципы

- Шаблон — контракт: не добавляй и не удаляй секции произвольно, заполняй существующие.
- Факты, а не догадки: probe-результаты, реальные URL/коды ответов, реальные пути.
- Acceptance Criteria и Gherkin должны быть **проверяемы** — без них задача не готова к старту.
- Обновляй `Updated`, `## Progress Log` и `Status` при каждом значимом изменении.
- См. также `docs/CODEX.md` (Backend task planning) и существующие задачи `001`–`009`
  как образцы.
