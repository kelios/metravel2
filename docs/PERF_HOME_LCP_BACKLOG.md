# Производительность главной (mobile) — статус и бэклог до 90+

/ (home), mobile PageSpeed/Lighthouse. Обновлено: 2026-06-09.

## Что уже сделано и задеплоено

Починен SSG-скелет (`scripts/ssg-skeletons.js`), коммиты `b8849d19` + `4ab400f8`:

1. **Parse-order баг (корневой).** Inline-скрипт удаления скелета живёт внутри
   `#ssg-skeleton`, который инжектится сразу после `<body>` — то есть ДО `<div id="root">`.
   При исполнении `getElementById('root')` был `null`, срабатывала ветка
   `if(!r){s.remove()}` и убивала скелет за ~30 мс, до того как он мог стать FCP/LCP.
   Скелет не работал ни на одном роуте (home/search/travel). Фикс: ждать
   `DOMContentLoaded`, если `#root` ещё не распарсен; home/search убираются по сигналу
   `app-hydrated`, travel — по своему `img[data-lcp]`.
2. **Реальный текст героя** в скелете главной вместо пустых пульс-баров (LCP-кандидат).
3. **CLS-фикс:** `#ssg-skeleton` → `position:fixed; inset:0` (вне потока, opaque bg,
   высокий z-index). `#root` остаётся на `top:0` → снятие оверлея не двигает контент.

## Почему LCP/SI/90+ не достигнуты — стена

Реальный LCP-элемент главной — `<div>` приложения (~32 535 px²), который рисуется
**только после гидратации (~10 с на mobile-троттлинге)**. Скелет даёт ранний кадр
(FCP 1.0 с), но не ускоряет появление реального контента: оно ждёт парсинга +
исполнения **eager-бандла ~4.4 MB** (`__common` 3.4 MB raw + `entry` 1 MB raw).

Lighthouse-диагностика главной:
- `unused-javascript`: **584 KB неиспользуемого на главной JS в `__common`** (из 693 KB),
  + 86 KB в `entry`.
- Main-thread: Script Evaluation **1809 мс**, Parsing/Compile 299 мс.

Никакой приём со скелетом это не обходит — нужно **резать eager-JS**.

## Бэклог до 90+ (приоритезировано) — НЕ сделано, риск архитектурный

Единственный реальный рычаг: уменьшить eager-бандл, который грузится на КАЖДОЙ
странице. Сборка уже хорошо code-split (128 чанков: quill, heic2any, pdf, Map и т.д.
ленивые) — проблема в шарящемся `__common`.

1. **Вынести Leaflet (~150 KB) из `__common`.** Сейчас он там, потому что
   `loadLeafletRuntime` (а значит `leaflet`/`react-leaflet`) шарится между
   `Map.web`, `WebMapComponent`, `TravelMap`, `QuestFullMap`, `UserPointsScreen`, и
   Metro хойстит общий async-vendor в общий чанк. Все динамические импорты УЖЕ на месте
   (`utils/loadLeafletRuntime.ts`) — нужна настройка сериализатора Expo/Metro, чтобы
   общий async-модуль шёл в отдельный чанк, а не в `__common`.
2. **Аудит остального «unused-on-home» в `__common`** (584 KB). Прод-бандл минифицирован
   без имён модулей — нужен билд с `ANALYZE_BUNDLE=1`, но `build-web-safe.js` при `-c`
   подменяет TMPDIR на временную папку и чистит её на выходе (дампы `metro-analyze-*`
   удаляются). Запускать `expo export` напрямую с `ANALYZE_BUNDLE=1` и стабильным TMPDIR,
   затем `node scripts/guard-eager-web-bundle.js --from-analyze --json`.
3. **Рассмотреть SSR реального hero-контента** (а не только скелета), чтобы LCP-элемент
   был в статическом HTML и не ждал гидратации. Большая архитектурная работа.

### Как верифицировать любую правку чанкинга (обязательно)
- `npm run build:web:prod`, затем проверить, что `__common` ужался и `leaflet` ушёл в
  отдельный чанк (`grep leaflet-control-layers-list dist/prod/_expo/static/js/web/*.js`).
- Карты работают на `/map`, `/travels/[slug]`, `/quests/[city]/[questId]`,
  `/userpoints` (Leaflet грузится по требованию, маркеры/тайлы рисуются).
- `npm run guard:eager-web:analyze` (бюджет eager ≤1200 KB).
- Повторный Lighthouse/PSI mobile на `/`.

### Риски
- Изменение сериализатора Metro/Expo может сломать ленивую загрузку карт на нескольких
  роутах. Команда раньше сознательно выбрала `inlineRequires` вместо кастомного чанкинга.
- Каждый прод-билд ~5–15 мин; цикл правка→билд→верификация дорогой.
- `build-web-prod.js` перезатирает `.env` копией `.env.prod` — после локальных прогонов
  восстанавливать из `.env.dev`.
