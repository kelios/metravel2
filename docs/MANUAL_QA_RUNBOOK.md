# Manual QA Runbook (Full Coverage)

Полный чеклист ручного прогона для web-версии `metravel`.

## 1. Цель и охват

Документ покрывает обязательные пользовательские потоки:
- auth entrypoints (регистрация/вход/выход);
- основной travel flow (create/read/update/delete);
- ключевые публичные страницы (home/search/map/roulette/travels);
- пользовательские разделы (favorites, history, profile, messages, subscriptions);
- legal/cookies/footer/navigation/SEO базового уровня;
- адаптивность desktop/mobile и базовая визуальная стабильность.

## 2. Подготовка перед прогоном

- Работать из корня репозитория (папка с `package.json`).
- Использовать тестовые доступы из `.env.e2e`.
- Не выводить секреты (`email`, `password`, `token`) в отчёты, логи, скриншоты и коммиты.
- Открыть новую сессию браузера (или очистить localStorage/cookies) перед guest-сценариями.
- Зафиксировать окружение прогона: ветка, коммит, `BASE_URL`, дата/время.

## 3. Приоритеты прогона

- `P0` — блокирующие релиз сценарии (обязательны всегда).
- `P1` — обязательны для feature/release веток, где менялись клиентские экраны/данные.
- `P2` — расширенный регресс и UX/визуальные проверки (по scope изменений).

## 4. P0: Критические сценарии (всегда)

### P0.1 Регистрация

Шаги:
1. Открыть `/registration`.
2. Ввести валидные имя, email, пароль, подтверждение пароля.
3. Нажать `Зарегистрироваться`.

Что проверить:
- Успешный результат (сообщение об успехе/подтверждении email).
- Нет зависших контролов после submit.
- Нет UI-ошибок, блокирующих повторный ввод.

### P0.2 Вход и выход

Шаги:
1. Открыть `/login`.
2. Войти под тестовым пользователем.
3. Перейти в меню аккаунта и выполнить выход.

Что проверить:
- После входа доступны приватные разделы (`/profile`, `/favorites`, `/history`).
- После выхода приватные разделы требуют авторизацию или показывают guest-state.
- Нет рассинхрона статуса (кнопка/имя пользователя соответствуют состоянию).

### P0.3 Создание путешествия (Create)

Шаги:
1. Войти в аккаунт.
2. Открыть `/travel/new`.
3. Заполнить обязательные поля (минимум: название; при необходимости описание).
4. Сохранить черновик или публикацию.

Что проверить:
- Сохранение проходит без ошибок.
- Созданная запись получает `id/slug/url` (явно или через переход на детали).
- Созданное путешествие можно открыть из списка или по прямой ссылке.
- Полный расширенный чеклист для P0.3: `docs/TRAVEL_CREATE_FULL_QA_CASE.md`.

### P0.4 Просмотр путешествия (Read)

Шаги:
1. Открыть страницу деталей созданного путешествия (`/travels/:slug` или `/travels/:id`).
2. Проверить ключевые секции страницы.

Что проверить:
- Страница деталей не падает в 404/500.
- Заголовок и контент соответствуют сохранённым данным.
- Основные блоки видимы: hero, описание, действия пользователя.

### P0.5 Редактирование путешествия (Update)

Шаги:
1. Перейти в `/travel/edit/:id` (или через кнопку `Редактировать`).
2. Изменить минимум одно поле (обычно название).
3. Сохранить и открыть детали повторно.

Что проверить:
- Изменения применились и сохраняются после перезагрузки.
- Не происходит редирект на логин для владельца.
- Нет потери ранее заполненных полей.

### P0.6 Удаление путешествия (Delete)

Шаги:
1. Открыть созданное путешествие/список своих путешествий.
2. Выполнить удаление (подтвердить диалог, если есть).
3. Проверить отсутствие записи в списках и по URL деталей.

Что проверить:
- Объект удалён, старый URL недоступен или показывает корректный not-found.
- Нет «битых» карточек/ссылок после удаления.
- Повторное удаление не ломает UI (если API отвечает 404 — корректно обработано).

## 5. P1: Основной функциональный регресс

### P1.1 Главная страница (`/`)

Что проверить:
- Загружаются основные секции и карточки/состояние пустого списка.
- Переходы из быстрых действий работают.
- Нет критичных layout-сдвигов в первый экран.

### P1.2 Список путешествий (`/travelsby`)

Что проверить:
- Карточки отображаются или виден корректный empty-state.
- Поиск/фильтры/сортировка применяются и сбрасываются.
- UI фильтров не ломается на повторном открытии/закрытии.

### P1.3 Поиск (`/search`)

Что проверить:
- Поле поиска фильтрует список.
- Очистка поиска возвращает исходный список.
- Нет подвисаний debounce/обновления результатов.

### P1.4 Карта (`/map`)

Что проверить:
- Карта и панель фильтров загружаются.
- Маркеры/кластеры интерактивны.
- Открытие popup и переход на детали работают.
- На mobile режимах panel open/close стабилен, без перекрытий.

### P1.5 Рулетка (`/roulette`)

Что проверить:
- Кнопка подбора маршрутов запускает выбор.
- После подбора отображается корректный результат/карточки.
- Обработка пустого ответа корректная.

### P1.6 Избранное и история (`/favorites`, `/history`)

Что проверить:
- Добавление в избранное отражается в списке избранного.
- Очистка истории/избранного работает и не ломает состояние страницы.
- Списки корректно отображают empty-state.

### P1.7 Профиль (`/profile`)

Что проверить:
- Страница профиля открывается для авторизованного пользователя.
- Обновление базовых полей сохраняется.
- Ошибки валидации отображаются без поломки формы.

### P1.8 Сообщения (`/messages`) и подписки (`/subscriptions`)

Что проверить:
- Для гостя показывается корректный auth-required экран.
- Для авторизованного пользователя страницы открываются.
- Переключение табов и базовые действия работают.

### P1.9 Footer + legal/cookies

Что проверить:
- Переходы footer на `/about`, `/privacy`, `/cookies` работают.
- Cookie banner показывается и корректно принимается.
- После принятия cookie баннер не блокирует сценарии.

### P1.10 Базовый SEO smoke (публичные страницы)

Что проверить:
- Есть валидный `title` и `description` на `/`, `/travelsby`, `/registration`.
- `canonical` не дублируется и соответствует странице.
- Для служебных/приватных страниц применяются корректные мета-ограничения (где предусмотрено).

## 6. P2: Расширенный регресс и UX

### P2.1 Адаптивность

Что проверить:
- Ключевые страницы корректны на desktop и mobile ширинах.
- Нет горизонтального скролла там, где его быть не должно.
- Нижний dock/footer не перекрывает контент.

### P2.2 Визуальная стабильность

Что проверить:
- Нет заметных CLS-сдвигов при первом рендере списков/деталей.
- Скелетоны корректно заменяются контентом.
- Placeholder изображений нейтральный и не ломает геометрию карточек.

### P2.3 Внешние ссылки

Что проверить:
- Открытие внешних ссылок работает без прямых runtime-ошибок.
- Навигация к внешним ресурсам не ломает возврат в приложение.

## 7. Минимальный набор для быстрого smoke (если мало времени)

Выполнить минимум:
1. `P0.1` Регистрация.
2. `P0.2` Вход/выход.
3. `P0.3` Create.
4. `P0.4` Read.
5. `P0.5` Update.
6. `P0.6` Delete.
7. `P1.4` Карта.
8. `P1.9` Footer + cookies.

## 8. Что фиксировать по каждому дефекту

- ID сценария (`P0.3`, `P1.4` и т.д.).
- Шаги воспроизведения.
- Фактический результат.
- Ожидаемый результат.
- URL и окружение (`BASE_URL`, ветка, коммит).
- Артефакт: скриншот/видео/лог консоли.
- Severity: `critical | major | minor`.

## 9. Критерий завершения ручного прогона

- Все `P0` пройдены без `critical` дефектов.
- Для текущего scope изменений пройдены релевантные `P1/P2`.
- Все найденные дефекты задокументированы с шагами и артефактами.
