# Backend workboard

Created: 2026-06-03.

Scope: issues that require changes in the **backend service** (separate repo / separate service — NOT this Expo frontend monorepo). Each item was surfaced by frontend QA / code review on the MeTravel frontend. The frontend already mitigates where possible; these tasks fix the root cause on the backend.

Rule: a frontend mitigation (URL normalization, treat-as-empty, client validation) is load-bearing — do NOT remove it on the frontend until the corresponding backend task here is shipped and verified, and never without a regression test.

Owner: backend team / Витаутас (DevOps) liaison. Source of truth for the frontend side stays in `docs/AGENT_WORKBOARD.md`.

Repro spec: `e2e/quest-video.spec.ts › should check video URL from API`. | `normalizeMediaUrl`/`fixMediaUrl` (`utils/mediaUrl.ts`) reconstructs the clean S3 URL; applied in `api/quests.ts` + `utils/questAdapters.ts`; covered by `__tests__/utils/questAdapters.fixMediaUrl.test.ts`. | Stop prefixing the API host when the stored value is already an absolute URL (guard `build_absolute_uri` on `^https?://`). Serializer should emit the raw S3 URL. |
| BE-002 | Comments endpoint returns 400/404 for travels with no comments | `F-002` | P2 | `GET /api/travel-comments/?travel_id={id}` returns **400 Bad Request** for travels with zero comments (observed `travel_id=391`, `563`). Empty comment set should be a normal 200 + `[]`, not an error status. | App treats the 400 as "no comments" and renders the empty state ("Пока нет комментариев"); comments read path no longer probes main-thread metadata (avoids the 404 console noise). | Return `200` + empty list (or paginated empty page) when a travel has no comments; reserve 400 for genuinely malformed `travel_id`. Confirm `/travel-comment-threads/main/` contract for the no-thread case. |
| BE-003 | `PUT /api/travels/upsert/` 500s on incomplete payload | `T-028` (PDF/export QA) | P2 | Minimal/empty payload to `PUT /api/travels/upsert/` → **500** with `KeyError 'id'` → `list index out of range`. | BE-004 | Places/map catalog endpoint has no server-side pagination | `PERF-019` (blocks `PERF-018`) | P2 | Frontend `fetchPlacesCatalog` (`api/places.ts:5`, `PLACES_CATALOG_PER_PAGE=1000`, radius `20000 км`, timeout 12s) pulls the **entire** catalog in one request on mount and normalizes all into memory. | BE-007 | User avatars served as full-resolution direct S3 URLs (no resized variants) | Image-optimization pass (`UserAvatar`, `ProfileHeader`, `settings`, `user/[id]`, messages, subscriptions) | P3 | `profile.avatar` / `userAvatar` arrive as direct object-storage URLs (`*.s3.amazonaws.com/…`; fixtures in `__tests__/context/AuthContext.test.tsx` show `https://…/avatar.webp?X-Amz-Expires=3600`, and `normalizeAbsoluteMediaUrl` in `utils/mediaUrl.ts:61` strips the `X-Amz-*` signature). These hosts are NOT own-domain and do NOT match the resize-proxy path regex `MEDIA_FILE_PATH` (only gallery, travel-image, travel-description-image, address-image), so `optimizeImageUrl` returns them untouched. Result: the header avatar (20–32px, loads on every page), profile header (124px), and message/subscription lists (40–56px) each download the full uploaded original. | Components already request a display-sized variant via `optimizeImageUrl(url, { width, height, quality, format: 'auto', fit: 'cover' })`; for non-own-domain S3 URLs this is a SAFE no-op (URL returned unchanged, presigned signature preserved). Bytes start dropping automatically once avatar URLs become proxy-eligible — do not remove these calls. | Deliver resized avatar variants, via one of: (a) generate stored avatar conversions (like gallery `…/conversions/…`) and return their URLs; (b) route avatar delivery through the same image-resize proxy that honors `?w=&h=&q=&f=&fit=&dpr=`; or (c) generate fixed-size webp variants (e.g. 48 / 96 / 248px) at upload time. Do NOT append query params to presigned S3 URLs — it breaks the signature. |

## Notes

- BE-001, BE-002, BE-003, BE-005, BE-007 are **safe to ship independently** — the frontend already degrades gracefully; the backend fix just removes the workaround's reason to exist.
- BE-007: the frontend `optimizeImageUrl` calls are already in place and are no-ops until avatar URLs become proxy-eligible — landing the backend variant delivery turns on the byte savings with zero further frontend work.
- BE-004 changes an API contract → must be coordinated with the frontend (`api/places.ts`, `api/map.ts`, `screens/tabs/PlacesScreen.tsx`) before/with `PERF-018`/`PERF-019`.
- BE-006 is infra/provisioning, not a code bug; it gates several authed QA tasks on the frontend board.
- When a BE task ships, update the linked frontend row in `docs/AGENT_WORKBOARD.md` and remove the corresponding mitigation **only with a regression test** guarding the new behaviour.
