## TL;DR

Документ собирает новые продуктовые идеи для MeTravel: геймификацию с бейджами и персонажами, админские награды, шаринг достижений, расширенное планирование совместных поездок, публичный каталог "Поехали со мной", коммуникацию участников, юридические согласия, trust & safety и privacy/security требования.

Это не готовое техническое задание для разработки. Перед реализацией каждый крупный блок нужно разложить на отдельные feature briefs, backend tasks, UI contracts и acceptance criteria.

## Scope

В документ входят:

- бейджи, медали, персонажи и прогресс;
- редкие награды, которые вручает администратор или модератор;
- шаринг медалей в соцсетях;
- планирование поездок и совместных выездов;
- публичные и приватные поездки;
- заявки на участие, нотификации и статусы;
- чат поездки и интеграции с мессенджерами;
- legal/disclaimer flow для мест, маршрутов, квестов и поездок;
- шифрование сообщений, защита контактов и настройки приватности.

Вне scope:

- точная схема БД;
- дизайн экранов;
- финальные юридические формулировки;
- выбор провайдера маршрутизации/GPX;
- production-ready модель модерации и anti-abuse.

## 1. Бейджи и геймификация

### Цель

Сделать активность пользователей заметной, эмоциональной и брендовой. Не просто показывать "очки", а дать пользователям понятные статусы, прогресс и коллекционные достижения.

### Бейдж за новое место

Выдавать пользователю отдельный престижный бейдж, если он добавил место, которого раньше не было на сайте.

В бейдже желательно показывать:

- дату открытия места;
- название места;
- сколько людей потом посмотрели, сохранили или посетили это место;
- ссылку на место;
- статус автора как первооткрывателя.

Возможные названия:

- Первооткрыватель;
- Следопыт;
- Разведчик;
- Первопроходец;
- Нашел первым;
- Открыл на карте;
- Зажег маяк.

### Линейки бейджей с прогрессией

Нужны эмоциональные, смешные и запоминающиеся названия, которые могут стать внутренним языком сообщества. Не использовать только стандартные "Новичок", "Эксперт", "Мастер".

Примеры линеек:

| Линейка | Уровни |
| --- | --- |
| Собачья | Щенок, Дворовый пес, Следопыт, Матерый пес, Пес Барбос, Легенда района |
| Кабанья | Поросенок, Кабанчик, Секач, Вепрь, Король леса |
| Лисья | Лисенок, Хитрюга, Рыжий следопыт, Старая лиса, Легенда нор |
| Птичья | Птенец, Воробей, Сокол, Орел, Белый орлан |

Идея бренда: выбрать один сильный маскот или несколько веток, чтобы пользователи могли говорить "я уже Барбос третьего уровня" или "у него легендарный Барбос".

### Персонажи вместо разрозненных бейджей

Вместо сотен независимых бейджей пользователь выбирает или развивает персонажа. Действия пользователя прокачивают выбранную ветку, а визуал персонажа меняется по мере роста.

Пример деталей прогресса:

- ошейник;
- рюкзак;
- компас;
- карта;
- медали;
- плащ путешественника.

### Типы активности

| Направление | Что прокачивает | Пример линейки |
| --- | --- | --- |
| Исследователь мест | добавление мест, подтверждение информации, фотографии, отзывы | Щенок, Следопыт, Разведчик, Первооткрыватель, Картограф, Легенда карты |
| Читатель | прочитанные статьи, сохранения, время чтения | Совенок, Любопытный, Книжный червь, Архивариус, Хранитель знаний, Великая Сова |
| Автор | статьи, комментарии, лайки на контент | Лисенок, Рассказчик, Хитрый Писарь, Летописец, Мастер Историй, Великая Лиса |
| Участник сообщества | лайки, реакции, ответы, помощь другим | Дружище, Свой человек, Душа компании, Главная капибара, Легендарная капибара |

### Визуализация профиля

Профиль должен показывать не только список бейджей, а прогресс и следующую цель.

Пример:

```text
Барбос 4 уровня
Прогресс: 80%

Исследователь мест
Читатель
Автор

Следующая цель:
Добавить еще 3 места
```

Возможен вариант нескольких персонажей:

```text
Барбос - уровень 7
Сова - уровень 5
Лиса - уровень 3
```

### Выбор пути развития

После достижения уровня пользователь может выбирать следующий путь:

```text
Вы достигли уровня "Следопыт".

Выберите путь:
- Картограф
- Разведчик
- Фотоохотник
```

Это превращает систему в легкую RPG-механику: пользователи сравнивают не только количество очков, но и свои "билды", ветки и роли в сообществе.

## 2. Админские и редкие награды

### Цель

Отделить автоматические достижения от редких наград сообщества, которые вручает человек. Такие награды ценнее, потому что их нельзя просто "нафармить".

### Общие правила

Редкие награды:

- выдает только админ или модератор;
- нельзя заработать автоматически;
- имеют ограниченное количество владельцев или высокую ценность;
- отображаются отдельно в профиле;
- имеют дату получения;
- имеют причину выдачи;
- должны попадать в журнал безопасности.

### Категории наград

#### Амбассадор MeTravel

Выдается за:

- упоминание сайта в соцсетях;
- публикацию роликов;
- привлечение новых пользователей.

Уровни:

- Друг MeTravel;
- Голос MeTravel;
- Амбассадор MeTravel;
- Легенда MeTravel.

#### Создатель историй

Выдается за контент про места с сайта:

- Shorts;
- Reels;
- TikTok;
- YouTube.

Возможные названия:

- Репортер;
- Охотник за историями;
- Полевой корреспондент;
- Легендарный рассказчик.

#### Первая волна

Для ранних пользователей, которые помогали запускать проект, тестировали и давали обратную связь.

Возможные названия:

- Первая волна;
- Основатель сообщества;
- Старожил;
- Был здесь до того, как это стало модным.

#### Сердце сообщества

Для пользователей, которые помогают другим:

- отвечают новичкам;
- помогают исправлять информацию;
- создают дружелюбную атмосферу.

Возможные названия:

- Добрый сосед;
- Сердце сообщества;
- Хранитель сообщества.

#### Легендарные награды

Для редких событий:

- пользователь открыл место, ставшее очень популярным;
- привел много пользователей;
- создал вирусный ролик про место из MeTravel.

Возможные названия:

- Живая легенда;
- Имя на карте;
- След в истории;
- Легенда MeTravel.

### Структура блока в профиле

Профиль можно разделить на три зоны:

- Персонажи и прокачка;
- Достижения;
- Редкие награды.

## 3. Шаринг медалей в соцсетях

### Цель

Дать пользователям возможность делиться медалями и редкими наградами. Это одновременно социальное доказательство и органический маркетинг MeTravel.

### User flow

У каждой медали есть действие:

```text
Поделиться
```

Пользователь может поделиться:

- в Instagram Stories;
- в Telegram;
- в Facebook;
- в TikTok / Shorts через скачиваемую картинку или видео;
- прямой ссылкой на профиль или достижение.

### Что генерировать

Для шаринга нужна карточка медали:

- название медали;
- ник пользователя;
- причина получения;
- дата получения;
- логотип MeTravel;
- ссылка на достижение или профиль;
- CTA: "Найди свое место на MeTravel" / "Смотри мой маршрут".

Пример текста карточки:

```text
Амбассадор MeTravel

Юля получила редкую медаль
за видео о найденном месте

MeTravel - места, которые находят свои люди
```

### Важно

Редкие админские медали должны шариться особенно красиво. Это бесплатная реклама и социальное доказательство, поэтому качество карточки должно быть выше, чем у обычных бейджей.

## 4. Планирование поездок и совместных выездов

### Цель

Расширить существующую механику планирования поездок: пользователь собирает маршрут из мест MeTravel, превращает его в событие и может пригласить других людей.

### Создание поездки

Кнопка:

```text
Запланировать поездку
```

Пользователь указывает:

- название поездки;
- описание;
- дату;
- время старта;
- способ передвижения: пешком, велосипед, мотоцикл, автомобиль, автодом;
- публичность: приватная или публичная;
- количество мест, если он готов взять участников;
- стартовую точку.

### Конструктор маршрута

Пользователь может:

- добавлять точки из MeTravel;
- менять порядок точек;
- удалять точки;
- добавлять собственные точки;
- добавлять точки отдыха;
- добавлять точки ночевки;
- сохранять шаблоны маршрутов.

### Автоматическое построение маршрута

После выбора точек система:

- определяет текущее местоположение пользователя, если он разрешил;
- строит маршрут;
- показывает расстояние;
- показывает примерное время в пути;
- показывает набор высоты, если доступно;
- показывает количество остановок.

### Экспорт маршрута

Поддержать:

- GPX;
- KML;
- Google Maps;
- Apple Maps;
- Garmin;
- Komoot.

Кнопки:

```text
Скачать GPX
Открыть в навигаторе
```

### Приглашение участников

Если у пользователя есть подписчики:

```text
Пригласить подписчиков
```

Если подписчиков нет или нужны внешние каналы:

```text
Поделиться поездкой
```

Каналы:

- Telegram;
- WhatsApp;
- Facebook;
- Instagram;
- прямая ссылка.

### Страница поездки

Страница поездки должна показывать:

- название;
- дату;
- количество точек;
- расстояние;
- организатора;
- количество участников;
- список участников или заявок;
- статус поездки;
- маршрут;
- CTA для участия.

Пример:

```text
Замки Беларуси

12 июля 2027
8 точек
350 км

Организатор: Юля
Участники: 12 человек
```

### Ответы участников

Приглашенный пользователь может выбрать:

- Поеду;
- Думаю;
- Не смогу.

### Совместное редактирование

Организатор может включить:

```text
Участники могут предлагать точки
```

После этого участник видит действие:

```text
Предложить место
```

Организатор принимает или отклоняет предложения.

### После поездки

Система предлагает создать отчет:

- посещенные места;
- фотографии;
- GPX-трек;
- участники;
- итоговая страница "Поездка состоялась".

После завершения:

- участники получают специальный бейдж;
- организатор получает опыт;
- посещенные точки сохраняются в профиле;
- маршрут может попасть в каталог готовых маршрутов.

## 5. Маршруты сообщества и экспедиции

### Маршруты сообщества

Новый тип контента: готовые путешествия пользователей, а не только отдельные места.

Примеры:

- Замки Беларуси за выходные;
- Лучшие водопады Польши;
- 50 км на велосипеде вокруг Варшавы;
- Заброшенные усадьбы Гродненской области.

### Экспедиция

В будущем можно ввести сущность "Экспедиция", которая объединяет:

- маршрут;
- участников;
- чат;
- фотографии;
- GPX-трек;
- отчет после завершения;
- достижения участников.

Это превращает MeTravel из каталога мест в платформу для организации реальных совместных поездок.

## 6. Коммуникация участников поездки

### Цель

После создания поездки организатор должен быстро собрать участников для общения. Если не дать встроенный путь, пользователи все равно будут вручную копировать ссылки и уходить в мессенджеры.

### MVP: Telegram-группа

На странице поездки:

```text
Создать Telegram-группу
```

Логика:

- организатор нажимает кнопку;
- система создает или помогает создать приглашение;
- участники получают ссылку;
- вступают самостоятельно.

### Привязка Telegram

В профиле пользователь может:

- указать Telegram username;
- авторизоваться через Telegram.

В поездке появляется действие:

```text
Пригласить в Telegram
```

### Автоматическое создание чата

При создании поездки можно добавить чекбокс:

```text
Создать Telegram-группу
```

Автоматически готовятся:

- название группы;
- ссылка-приглашение;
- описание поездки.

Пример:

```text
Замки Беларуси - 12 июля

Организатор: Юля
Маршрут: 8 точек
```

### Другие платформы

В будущем поддержать:

- Telegram;
- WhatsApp;
- Discord;
- Facebook Messenger;
- Signal.

В профиле можно указать предпочитаемый мессенджер.

### Контакты и приватность

Разделить публичное участие и раскрытие контактов:

- пользователь может быть виден в списке участников;
- контакты не раскрываются автоматически;
- организатор может отправить запрос "Запросить контакт";
- пользователь подтверждает или отклоняет.

### Временный чат поездки

Чат живет вместе с поездкой.

Статусы:

- Планируется;
- Активна;
- Завершена.

После завершения:

- чат архивируется;
- фотографии остаются;
- обсуждение может стать частью истории маршрута.

## 7. Публичные поездки "Поехали со мной"

### Цель

Создать каталог публичных поездок, где пользователи могут объявлять совместные выезды и принимать заявки от других участников.

### Тип поездки

При создании поездки пользователь выбирает:

- Приватная: доступна только по ссылке или приглашению;
- Публичная: видна в общем каталоге поездок.

### Каталог

Отдельная вкладка:

```text
Поехали со мной
```

Карточка публичной поездки показывает:

- куда едем;
- дату и время;
- стартовую точку;
- свободные места;
- способ передвижения;
- организатора;
- статус: открыта, мест нет, завершена.

### Заявка на участие

Пользователь нажимает:

```text
Хочу поехать
```

Дальше он должен:

- написать короткое сообщение организатору;
- указать соцсети или подтвердить привязанные соцсети;
- согласиться с правилами безопасности;
- подтвердить понимание, что MeTravel не организует поездку.

### Безопасный flow

1. Пользователь подает заявку.
2. Общение сначала идет внутри MeTravel.
3. Организатор смотрит профиль, активность, соцсети и бейджи.
4. Организатор принимает, отклоняет или задает вопрос.
5. Только после approval можно открыть контакты или соцсети, если обе стороны согласны.

### Статусы заявки

- Новая;
- На рассмотрении;
- Одобрена;
- Отклонена;
- Отменена.

### После одобрения

Участнику открывается:

- точное место встречи;
- чат поездки;
- дополнительные контакты, если организатор разрешил;
- детали маршрута;
- список участников.

### Нотификации

Когда кто-то хочет присоединиться, организатор получает:

- уведомление на сайте;
- email;
- возможно Telegram-уведомление, если подключено.

Пример:

```text
Анна хочет присоединиться к поездке "Замки Беларуси".
Посмотреть заявку
```

## 8. Trust & Safety

### Цель

Снизить риски в сценариях, где пользователи встречаются, обмениваются контактами или участвуют в поездках друг друга.

### Меры доверия

Нужны:

- жалоба на пользователя;
- блокировка пользователя;
- рейтинг или отзывы после поездки;
- подтверждение email;
- подтверждение Telegram;
- подтверждение Facebook;
- подтверждение Instagram;
- история активности на сайте;
- бейдж "Проверенный участник";
- статус "Организатор с опытом".

Важно: подтверждение аккаунта не является гарантией личности пользователя.

### Предупреждение о личных договоренностях

В местах, где пользователь вступает в поездку, обменивается контактами или идет во внешний чат, явно показывать:

```text
MeTravel не несет ответственность за личные договоренности между пользователями.
```

## 9. Legal и informed consent

### Цель

Пользователь должен понимать, что информация на сайте может быть неполной или устаревшей, а участие в поездках, квестах и маршрутах происходит на его риск.

Финальные формулировки обязательно должны пройти юридическую проверку.

### Обязательные страницы

Нужны отдельные страницы или документы:

- Отказ от ответственности (Disclaimer);
- Пользовательское соглашение;
- Правила сообщества;
- Правила участия в поездках и мероприятиях.

## 10. Security & Privacy

### Цель

Функции чатов, совместных поездок и обмена контактами требуют отдельной privacy/security модели. Пользовательские сообщения и контакты нельзя хранить и раскрывать как обычный публичный контент.

### Шифрование личных сообщений

Требование:

- все личные сообщения должны храниться в зашифрованном виде;
- при компрометации базы данных злоумышленник не должен иметь возможность прочитать переписку пользователей;
- не хранить сообщения в открытом текстовом виде.

MVP:

- шифрование сообщений на сервере;
- ключи хранения отдельно от базы данных;
- резервные копии также шифруются.

Будущее:

- End-to-End Encryption для личных сообщений;
- End-to-End Encryption для групповых чатов.

### Шифрование чувствительных данных

Шифровать или дополнительно защищать:

- email;
- телефоны, если появятся;
- ссылки на соцсети;
- личные контакты;
- приватные заметки пользователей.

### Минимизация доступа администрации

Принцип:

```text
Администратор не должен видеть данные пользователя без явной необходимости.
```

Разделять:

- контент сайта;
- служебные данные;
- личные данные пользователей.

### Защита контактов

До подтверждения заявки на поездку:

- Instagram скрыт;
- Facebook скрыт;
- Telegram скрыт;
- другие контакты скрыты;
- показывается только профиль MeTravel.

Контакты раскрываются только после:

- взаимного согласия;
- или одобрения организатором, если такая модель будет выбрана.

### Журнал безопасности

Логировать:

- входы;
- смену пароля;
- привязку соцсетей;
- выдачу админских медалей;
- раскрытие контактов;
- действия модераторов.

### Настройки приватности

Пользователь должен выбирать, кто видит:

- мои поездки;
- мои маршруты;
- мои соцсети;
- мои достижения;
- мой список посещенных мест.

Варианты видимости:

- Все;
- Только зарегистрированные;
- Только подписчики;
- Только я.

### Раскрытие контактов

Перед раскрытием контактов показывать:

```text
Контакты будут переданы другому пользователю.

Убедитесь, что готовы продолжить общение вне платформы.
```

### Пользователь владеет своими данными

Долгосрочно нужны:

- экспорт своих данных;
- удаление аккаунта;
- удаление переписки;
- удаление маршрутов;
- отзыв согласий;
- прозрачные настройки приватности.

## MVP proposal

Минимальный полезный запуск можно разделить на независимые этапы.

### MVP 1: бейджи

- Бейдж за добавление нового места.
- Простая модель достижений в профиле.
- Базовые уровни без сложных веток.
- Админская выдача одной-двух редких наград.
- Журнал выдачи админских наград.

### MVP 2: шаринг достижений

- Кнопка "Поделиться" на медали.
- Генерация статичной карточки.
- Прямая ссылка на профиль или достижение.
- Telegram/Facebook/native share, где поддерживается.

### MVP 3: планирование поездки

- Расширенная страница поездки.
- Ручное добавление точек маршрута.
- Публичная/приватная поездка.
- Инвайт по ссылке.
- Ответы "поеду / думаю / не смогу".

### MVP 4: публичные поездки

- Вкладка "Поехали со мной".
- Заявка на участие.
- Статусы заявки.
- Нотификации на сайте и email.
- Обязательные consent чекбоксы.

### MVP 5: коммуникация и privacy

- Встроенный чат поездки или ссылка на Telegram-группу.
- Контакты скрыты до approval.
- Предупреждение перед раскрытием контактов.
- Серверное шифрование личных сообщений.

## Affected areas

Вероятные зоны будущей реализации:

- `screens/` and `app/` - новые страницы профиля, достижений, поездок и публичного каталога;
- `components/` - карточки достижений, медалей, поездок, заявок, consent UI;
- `api/` and `services/` - достижения, маршруты, заявки, нотификации, чат, privacy settings;
- `hooks/` - focused hooks для профиля, поездок, заявок и чатов;
- `utils/externalLinks.ts` - все внешние переходы в соцсети и мессенджеры только через project-approved external-link chokepoint;
- `constants/designSystem.ts` - дизайн-токены для новых UI-состояний;
- MCP task board - frontend/backend задачи через `ticket-board` по правилам `docs/TASK_BOARD_MCP.md`.

## Validation notes

Перед разработкой отдельных фич:

- для UI изменений подключить `$metravel-ui-guardrails`;
- для backend задач создать board tasks через `ticket-board` с Task Contract и `area=back`;
- для external links выполнить `npm run guard:external-links` или `npm run governance:verify`;
- для visible web UI пройти browser preview, screenshot и console check;
- для security/privacy решений сделать отдельный architecture review.
