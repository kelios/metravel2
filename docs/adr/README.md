# Architecture Decision Records

ADR — короткая фиксация архитектурных решений: контекст → решение → последствия.
Нужны чтобы через полгода понять "почему так сделано" без раскопок в git blame.

## Когда писать ADR

- Выбор между альтернативами, где оба варианта рабочие (TanStack Query vs SWR, Zustand vs Redux).
- Жёсткое правило, которое будут нарушать без аргументации (запрет прямого `Linking.openURL`, обязательность `UnifiedTravelCard`).
- Отказ от очевидного пути с неочевидной причиной.

## Когда НЕ писать ADR

- Очевидные решения (используем TypeScript, у нас есть тесты).
- Мелкие стилистические выборы.
- Документация фичи — для этого `docs/features/`.

## Процесс

1. Скопируй `docs/adr/TEMPLATE.md` → `docs/adr/NNNN-kebab-case-title.md` (NNNN — следующий номер).
2. Заполни. Держи под 150 строк.
3. Статус ставь `Accepted` если уже применяется, `Proposed` если на обсуждении.
4. Отменяешь ADR — не удаляй, пометь `Superseded by NNNN` и добавь новый.
