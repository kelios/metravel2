# Travel Create: Full QA Case (P0.3)

Полный тест-кейс для проверки создания путешествия (`/travel/new`) и последующей валидации, что данные корректно сохраняются, отображаются и фильтруются.

## 1. Цель

Подтвердить, что пользователь может:
- создать путешествие во всех основных режимах;
- заполнить все ключевые поля без потери данных;
- работать с картой/координатами, галереей, плюсами/минусами;
- сохранить черновик/обновления и открыть результат в деталях;
- найти созданное путешествие через фильтры/поиск/карту (если опубликовано).

## 2. Предусловия

- Есть авторизованный тестовый пользователь.
- Доступны тестовые данные из `.env.e2e` (не логировать секреты).
- Приложение доступно по локальному `BASE_URL`.
- Браузер: desktop + mobile viewport (минимум один прогон на каждый).

## 3. Покрываемые режимы

1. Полный мастер создания (все шаги).
2. Quick draft (быстрый черновик).
3. Редактирование после создания (`/travel/edit/:id`).
4. Повторное открытие черновика (draft recovery / autosave).
5. Работа с картой и fullscreen-режимом (если включён в текущем UI-билде).

## 4. Основной сценарий (Happy Path, полный мастер)

1. Открыть `/travel/new`.
2. Заполнить обязательные поля:
   - название;
   - краткое описание;
   - базовые параметры поездки.
3. Добавить маршрут/точки на карте.
4. Добавить медиа в галерею.
5. Заполнить расширенные параметры (категории, транспорт и т.д.).
6. Сохранить как черновик.
7. Открыть созданное путешествие в деталях.
8. Перейти в редактирование, изменить часть полей, сохранить.
9. Повторно открыть детали и убедиться, что изменения применились.

Ожидаемый результат:
- Создание и редактирование проходят без ошибок.
- Данные не теряются между шагами, после перезагрузки и после повторного входа.

## 5. Матрица полей и проверок

### 5.1 Базовые поля

- `name` (название):
  - проверка min/max длины;
  - проверка валидации на пустое значение;
  - отображение в карточке и деталях.
- `description`:
  - длинный текст, переносы, спецсимволы;
  - корректный рендер в деталях.
- `year`, `number_days`, `number_peoples`, `budget`, `visa`:
  - сохранение значений и повторное отображение в edit/details.

### 5.2 Таксономии и фильтруемые поля

- `categories`, `transports`, `complexity`, `companions`, `month`, `over_nights_stay`, `country/city`:
  - выбор одиночных и множественных значений;
  - сохранение в API;
  - отображение на деталях;
  - фильтрация в списке `/travelsby`, `/search`, `/map` (для опубликованной записи).

### 5.3 Контентные доп. поля

- `plus` / `minus`:
  - заполнение, сохранение, рендер в деталях.
- `recommendation`:
  - сохранение и отображение.
- `youtube_link`:
  - валидный URL, невалидный URL (валидация), рендер блока.

### 5.4 Координаты, карта, fullscreen

- Добавление точек маршрута:
  - ручной ввод координат;
  - выбор точки через карту;
  - изменение/удаление точки.
- Линия маршрута:
  - формируется после добавления точек;
  - не ломается после сохранения и повторного открытия.
- Fullscreen-режим карты:
  - переход в fullscreen и обратно;
  - сохранность текущего состояния (точки, зум, выбранный слой);
  - отсутствие перекрытий UI-элементов.

### 5.5 Галерея и медиа

- Upload изображения:
  - успешная загрузка;
  - отображение превью;
  - сохранение после submit.
- Удаление изображения:
  - удаление из UI;
  - удаление сохраняется после повторного открытия.
- Невалидное/битое изображение:
  - корректный fallback без краша;
  - нет ломки layout.

## 6. Негативные сценарии (валидации и ошибки)

1. Сохранение без названия.
2. Слишком короткое название.
3. Невалидный YouTube URL.
4. Пустой/частично заполненный маршрут.
5. Сбой API на сохранении (проверка error-state + retry).
6. Потеря сети во время autosave.

Ожидаемый результат:
- Понятные сообщения об ошибках.
- Нет silent-fail.
- Пользователь может продолжить после исправления/повторной попытки.

## 7. Проверки после создания (обязательно)

После успешного create/update проверить:

1. Детали (`/travels/:slug`):
   - видны все заполненные поля;
   - корректные title/meta/canonical.
2. Списки:
   - запись появляется в соответствующих списках (для publish=true);
   - карточка открывается без ошибок.
3. Поиск и фильтры:
   - находится по названию;
   - фильтруется по выбранным параметрам.
4. Карта:
   - если есть координаты, точка/маршрут отображается на карте.
5. Избранное/история:
   - при открытии деталей запись попадает в историю;
   - избранное работает без рассинхрона.

## 8. Критерии приёмки P0.3

- Нет `critical`/`major` дефектов в create/edit flow.
- Все обязательные поля сохраняются и читаются обратно без искажений.
- Карта/координаты/галерея работают стабильно.
- Опубликованное путешествие корректно участвует в поиске и фильтрации.

## 9. Автоматизация (какие e2e тесты должны покрывать кейс)

- `e2e/travel-full-flow.spec.ts`
- `e2e/travel-persistence.spec.ts`
- `e2e/draft-recovery.spec.ts`
- `e2e/gallery-delete-broken-image.spec.ts`

## 10. Формат отчёта по прогону

Для каждого шага фиксировать:
- статус (`PASS/FAIL`);
- URL;
- входные данные;
- фактический результат;
- ожидаемый результат;
- артефакт (скрин/видео/trace);
- severity при дефекте.
